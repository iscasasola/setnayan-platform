## 2026-07-21 · fix(pricing): retiring a SKU actually stops the sale + Pro vendors regain payment links

Two live defects surfaced by the 2026-07-21 sell-vs-deliver catalog audit. Both are the same shape: a gate reading the wrong source of truth, so an owner-made decision in `/admin/pricing` silently did nothing.

**1. Retirement was a no-op — retired SKUs stayed fully purchasable.** `submitOrderAction` never required a `serviceKey` to map to an ACTIVE catalog row, so `COUPLE_WEBSITE_PRO` (₱4,999) and `INDOOR_BLUEPRINT` (₱1,499) — both `is_active=false` in prod — still rendered a live buy drawer and still created orders. Website PRO is the worse one: its two surviving perks now sell standalone for ₱3,998, so the retired umbrella overcharged by ₱1,001. Migration `20270712300000` had tried to stop this and its own comment concedes "this only stops new sales" — it didn't.

- New `resolveServiceSellability()` (`lib/v2-catalog.ts`) — a probe returning `sellable | retired | unknown | error` across both `platform_retail_catalog_v2` and `platform_package_catalog`.
- `submitOrderAction` (`dashboard/[eventId]/checkout/actions.ts`) rejects `retired` **before** either charge resolver runs. This replaces the hardcoded `GUIDED_PACK`/`MEDIA_PACK` denylist, which the generic probe subsumes exactly (both are `is_active=false` package rows; both live `PAPIC_UNLOCK*` bundles stay sellable).
- `studio/indoor-blueprint/page.tsx` no longer falls back to a compiled-in `INDOOR_BLUEPRINT_PRICE_PHP = 1499`; with no readable active price it renders no price and no drawer (constant deleted from `lib/indoor-blueprint.ts`).

⚠ **The obvious fix here is backwards and was deliberately NOT taken.** Adding `.eq('is_active', true)` to `resolvePaxPricedOrderCentavos` looks right but is strictly worse than the bug: in `submitOrderAction` a null resolve does not reject — it keeps the **client-supplied** `original_centavos` (the file says so itself: "Only SKUs in NEITHER catalog … keep the client value"). Filtering there would downgrade a retired SKU from *"charged its real ₱4,999"* to *"charged whatever the browser POSTs, including ₱1."* Hence a reject before the resolvers, never a filter inside them. `formatV2Sku` is likewise left unfiltered — after this guard every drawer it fronts is already closed, and filtering it would blank owned customers' price rows.

⚠ **`is_active` is overloaded, and a blanket "make every reader honor is_active" sweep is unsafe.** On `SETNAYAN_AI_RENEW` (₱799) the flag means *"not independently sellable"*, not *"retired"* — it is the live renewal price past cycle 1. `lib/setnayan-ai-event-pricing.ts` reads it UNFILTERED on purpose. (The audit's claim that filtering it would cause an 88% overcharge was checked and is **wrong** — `setnayanAiEventPricing` coerces each side against its own constant, so a missing renewal row falls back to ₱799, the correct price. The real risk is silent drift from the catalog, not a mischarge. Recorded here so the milder truth is what survives.)

**2. Pro/Enterprise vendors could never add a payment link.** `isVendorProActive` (`lib/vendor-payment-methods.ts`) gated on `orders.service_key IN ('vendor_pro_weekly','all_tools_unlock_annual')` — two V1 SKUs retired 2026-05-28. Prod has **zero** orders with either code ever, while five `vendor_profiles` sit at `tier_state='pro'`; the subscription RPCs never write to `orders` at all. Every paying Pro vendor hit an infinite "upgrade to Pro" loop on a feature they already owned. Now reads `vendor_profiles.tier_state` via the canonical `isTierAtLeast`, matching every other vendor gate. Two prod realities pinned by tests: `tier_expires_at IS NULL` means *no expiry* (all five real Pro rows), and one user owns 46 profiles so any-row-passes is required.

New tests (`pnpm test:unit`, 2449 passing): `lib/service-sellability.test.ts` (incl. design pins that `unknown` must ALLOW — `PAPIC_CAMERAS`, `SETNAYAN_AI_SUB`, `save-the-date:<slug>` and `vendor_additional_branch__<uuid>` legitimately have no catalog row — and that a read error fails CLOSED), `lib/setnayan-ai-renewal-price.test.ts`, `lib/vendor-payment-methods.test.ts`.

No migration.

**Known gap, deliberately NOT in this PR:** the Papic camera ladder bypasses `submitOrderAction` entirely — `studio/papic/actions.ts` has three direct `orders.insert` calls under the synthetic key `PAPIC_CAMERAS`, which is in neither catalog and so classifies `unknown`/allow forever. `fetchCameraRates` reads its four rate SKUs unfiltered and `PAPIC_CAMERA_LTD_DAY` (₱50) is `is_active=false` in **both** the catalog and `papic_tier_config` yet still offerable from a hardcoded rung list. Fixing it removes a visible rung from the highest-volume buy path, so it needs an owner call first: **is Ltd a live rung (flip it active) or retired (drop it)?**
