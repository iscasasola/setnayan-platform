## 2026-07-21 · refactor(admin): remove the Add-ons tab and the bundle maker — two fake doors on the pricing surface

Owner: *"we do not need the bundle maker as well"* — following the Add-ons-tab recommendation in the pricing-cleanup audit. Both surfaces removed here share a shape: they looked like working admin tools but could not do the thing they advertised.

**1. The Add-ons tab (`_surfaces/addons-surface.tsx`, 638 L) — broken three ways.** It was the last surface reading the legacy v1 `service_catalog`, and measured against prod it rendered exactly **17 rows, every one `is_active=false`** — it structurally could not display the one active row (`vendor_verification_initial`), because its category filter never included that category. Its "purchases" counter joined v1 `sku_code` against `orders.service_key`, which holds **V2** `service_code`s, so all 17 cards read *"No purchases yet"* permanently (the single legacy code in `orders` is a `cancelled` `save_the_date_video`, which the paid/fulfilled filter excludes anyway). Its eligibility dots joined `sku_code` against `feature_policy.feature_key`, which holds slugs (`papic`, `panood`, `led`) — a **zero-intersection join**, so every dot was hollow with an amber *"No policy set"*. Catalog Studio is now four tabs.

**2. The "Create a bundle" card + its `createBundle` action (~80 L).** It wrote a `platform_package_catalog` row from a name + price — but a bundle is **inert without `bundle_components` rows** saying which SKUs it unlocks, and the maker had no way to write those. The page's own footer admitted the gap: *"defining which services a bundle unlocks is a separate follow-up."* So every bundle it produced would render as a priced, sellable-looking product on `/pricing` that granted **nothing**. Prod confirms it was never used: zero `createBundle` audit rows (the one `%bundle%` entry is an *edit*), and all four real bundles are migration-authored with their components.

**The two live bundles stay fully editable.** `PAPIC_UNLOCK` ₱15,000 (7 components) and `PAPIC_UNLOCK_LTD` ₱9,000 (2 components) are referenced across ~20 live files — checkout, Papic studio, entitlements, onboarding — so only the CREATE path is gone, not the Bundles section. Its blurb now states plainly that membership is `bundle_components` and migration-owned, so a new bundle is authored in a migration alongside its components.

**Nothing orphaned.** Per the wayfinding rule, the Add-ons tab's one worthwhile piece — the legacy-catalog report export at `/admin/addons/pricing-report`, the only download path for retired v1 rows — is **kept** and its button re-homed onto the Pricing tab header as *"Download legacy catalog report"*. `/admin/addons` still resolves: the stub now redirects to `/admin/pricing` (the dead `?sku=` forwarding dropped). Route path unchanged so bookmarks survive. Nav cleaned in six places: `admin-nav-groups.tsx`, `admin-bottom-nav.tsx`, `admin-nav-descriptions.ts`, `lib/route-meta.ts`, `lib/routes.ts` (keeps `pricingReport`, drops the dead tab `index`), `lib/nav-registry-defaults.ts`. Verified `nav_slot_override` in prod is empty, so no stored override points at the removed row and no migration is needed.

Also fixed a dangling link the audit missed: `admin/studio/_surfaces/patiktok-surface.tsx` had a *"Back to add-ons"* link to `/admin/pricing?tab=addons`. It would not have 404'd — `coerceTab` silently swallows an unknown tab and falls back to Pricing — which is exactly why it was worth catching rather than leaving as a link whose label lies about where it goes. Now *"Back to pricing"* → `/admin/pricing`.

No DB rows deleted, no migration, no schema change, no customer-visible change. Typecheck + lint clean.
