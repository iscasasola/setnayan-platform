## 2026-06-29 · feat(papic): walk-up self-register foundation — scan the event QR, get a camera (no roster)

First code landing of the Papic walk-up plan (`0012_papic/Papic_Walkup_Face_Identity_Plan_2026-06-29.md` §1, §5). Until now a Papic guest camera required a pre-existing roster guest (the `setnayan_guest_session` cookie carries a `guest_id` you only get by opening a personal invite). This adds the "self-register on scan" entry — no guest list, no name.

- **Migration `20270321802042`** (schema-first): `events.papic_walkup_token` — a **dedicated guest-facing** walk-up secret (+ rotation clock), deliberately separate from the privileged crew `master_qr_token` (rotating the crew QR must not break guest cameras; the guest QR must not leak crew-device pairing). Plus `guests.self_registered` (marks walk-up cameras so the host roster can filter them + cleanup can target them) + `papic_walkup_register(p_walkup_token)` — a SECURITY DEFINER, anon-callable RPC that resolves the event by `papic_walkup_token`, **requires the event to own `PAPIC_GUEST`** (reuses `papic_event_owns_service`), inserts a lightweight nameless guest, and returns `{guest_id, event_id, qr_token}`. Idempotent, purely additive.
- **Route `GET /papic/join/[token]`**: resume-or-create. Gates on `eventPapicGuestActive` (the **same** alias/bundle-aware check the capture surface uses, so walk-up is available exactly when `/papic/guest` is — `eventSkuActive('PAPIC_GUEST')` = a `PAPIC_GUEST` order OR the `PAPIC_UNLOCK` bundle; the RPC repeats a DB backstop). A valid cookie already bound to this event → reuse the camera (fixes "re-scan makes a new camera"); otherwise mint a walk-up guest via the RPC, set the cookie on the redirect, and drop the guest at `/papic/guest`. Failures fall back to its existing empty states.
- **`lib/guest-session.ts`**: extracted `buildGuestSessionCookie()` (shared cookie spec) so the Route Handler can attach the session cookie directly to the redirect response; `setGuestSession()` now uses it.
- **Host surface**: `WalkupQrCard` on the Papic studio (`dashboard/[eventId]/studio/papic`, beside the off-roster "add a camera that isn't on the guest list" section) renders the scannable walk-up QR + copyable `/papic/join/<token>` link via `lib/qr.ts` `renderUrlQrSvg`. Shown only when `guestCamerasActive` (`ownsPapicUnlock || eventPapicGuestActive`), so it never offers a QR that wouldn't work. This makes the walk-up reachable end-to-end: host shows the QR → guest scans → camera.
- **Saved-link re-entry** ("save my camera"): `GET /papic/resume/[token]` re-establishes the guest-session cookie from a walk-up guest's own `qr_token` (restricted to `self_registered` guests; roster guests use `/papic/me` + their invite), gated like the join route, → `/papic/guest`. The capture surface (`papic-guest-capture.tsx`) shows a slim "Save your camera" strip (the `/papic/resume/<qr_token>` link + copy) **only for walk-up guests**, so they can return cross-device / after a cleared cookie — until the face engine adds face re-entry. Cookie-resume (60d) still covers the same-browser case.

Scope = end-to-end walk-up (host QR → scan → resume-or-create → capture) **+ saved-link cross-device recovery**. Remaining follow-ups: cross-device **face** re-entry (face engine), the consent/enroll step (face phase), and the first-5-free walk-up free tier (pricing — owner). Stays entirely in the Papic lane (`app/papic/*`, `studio/papic`, `guest-session.ts`) — no overlap with the parallel "Vids AI" Stories session.
