## 2026-07-17 · fix(creator): lock inquiry-provenance columns + block creator self-referral

Closes the two attribution-integrity gaps an adversarial review of merged PR #3340 (Creator Economy PR-C) found in the "inquiries driven" metric. No money-logic touched — token burns are sound; this only makes the metric non-forgeable.

**G1 — provenance columns were writable by any thread party.** PR #3340 (migration `20270819553697`) added `referring_chapter_id` / `inquiry_source` / `is_returning` to `chat_threads` with no write protection. The base RLS policy `chat_threads_member_write` is `FOR ALL TO authenticated`, so any thread party (couple OR vendor) could PATCH those columns via PostgREST — forging `referring_chapter_id` to any existing chapter (the FK only checks existence), inflating the public count, firing a false `chapter_drove_inquiry` notification, and mislabeling the vendor's real token burn as `spend_source='lead_unlock'`.
- `supabase/migrations/20270820292403_guard_thread_provenance_columns.sql`: BEFORE UPDATE trigger `guard_thread_provenance_columns` that reverts the three columns to their OLD value for any NON-privileged caller (privileged = `auth.role()` NULL / `service_role`, or `is_admin()`). Mirrors the house pattern `guard_users_privilege_columns` (`20270814328403`) + `guard_pax_finalize_columns`. **Chosen over a bare column-level `REVOKE UPDATE (…)`** because Supabase's platform baseline grants TABLE-LEVEL UPDATE to `authenticated`, and PostgreSQL checks table-level UPDATE before column grants — so a bare REVOKE is a no-op unless every other column is re-granted (fragile; silently re-opens on any new column). Verified the legit path is UNBROKEN: the only writer of these columns is `stampThreadProvenance` (`lib/inquiry-attribution.ts`), which runs on the SERVICE-ROLE admin client (`createAdminClient`); the thread upsert in `startServiceInquiry` never sets them, so the guard is a no-op for the ordinary inquiry flow.

**G2 — a creator could self-refer through their own chapter's Book CTA** and tick their own "inquiries driven" count. `startServiceInquiry` never checked `referral.creatorUserId !== user.id`.
- `apps/web/app/v/[slug]/inquiry-actions.ts`: when the resolved referral's creator IS the inquirer, DROP the referral to null (degrade to a normal Website inquiry — never error the inquiry). Also reject a client-supplied `inquiry_source` of `'influencer'`/`'degree'` (server-derived only).
- `apps/web/lib/inquiry-attribution.ts` `fetchInquiriesDrivenForCreators`: defense-in-depth — skip any attributed thread whose `created_by_user_id` equals the chapter's creator, so a legacy/forged self-referral never counts even if it slipped past the action-level drop.

**Test:** `apps/web/tests/db/creator-loop.db.test.ts` gains a case (executed against the real replayed schema): a service-role first-stamp lands; a thread party's PATCH of the three provenance columns is reverted while a non-guarded column edit still succeeds (proving the trigger — not RLS — protects the columns).

SPEC IMPACT: None (security hardening of already-shipped PR #3340 / Creator Economy PR-C; DECISION_LOG.md row appended in the spec corpus). References the PR-C money-path review G1/G2.
