## 2026-07-16 · feat(seating): 3D manipulation parity — move/rotate validate through the shared oracle + Save & view switch

The final phase of the seat-plan program (after the placement oracle #3277 and lock-gated vendor presence #3281). The 3D lab already moved and rotated tables and persisted through the same server actions as the 2D editor — but its drop-time validation ran on a **forked, disc-based** `checkPlacement` (bounding circles, a 3D-only stage/dance/sweetheart rule, no walkway, no monotone-escape). So 2D and 3D could disagree about the same drop. This wires the lab's move + rotate onto the ONE placement oracle (`checkPlacement` / `penetrationDepth` in `lib/seating.ts`, #3277), so outcomes are identical by construction — no 3D-specific geometry left in the manipulation path.

- `app/dashboard/[eventId]/seating/lab/_components/seating-lab-3d.tsx`:
  - **Oracle bridge (metres).** New `oraclePose` / `scaleOfTable` / `oracleZones` / `rotationBlocked` mirror the 2D editor's `poseAt` / `scaleOf` / `zonesFor` / `rotationBlocked` with `pxPerMeter` folded out to 1. The oracle is scale-homogeneous (it only compares lengths), so a metre-space verdict is bit-identical to the 2D pixel-space verdict — every length there is this one × `pxPerMeter`. All shape geometry stays in the shared `obbOf`/`checkPlacement`. Gated to a sized (walled) room; the free auto-grow board stays place-anywhere in both projections.
  - **Move = monotone-escape.** `onFloorMove` now runs the 2D editor's § 4 slide-and-non-worsening rule per drag frame (refs only, no React churn): a clean table stays fully legal (axis-separated slide glides it to the next gap); a table that starts overlapping may move only to a valid pose or one no deeper than ε = 2 cm. A dragged table can no longer be dropped through another table, the dance floor, the cocktail room, or a vendor booth, and the walkway (0.6 m default, matching the 2D aisle) is kept clear. The serpentine tip-snap pre-check (a sanctioned join → links on drop) still wins ahead of the escape, mirroring the 2D weld-first ordering.
  - **Rotate = preview→check→commit.** `rotateSelected` validates the target angle through the oracle before it commits (same as the 2D `commitRotation`) — a twist with no room is refused with an honest notice; same-`link_group_id` members are exempt (the weld model).
  - **Group move** validated through the same oracle (body overlap vs non-members + zones) at commit; the rigid unit still moves as one.
  - **Visual feedback:** the dragged table wears a subtle **gold** ground ring while its pose is oracle-valid, a warm-**red** ring while it's escaping (colour driven from a ref inside `useFrame`).
  - The forked disc `checkPlacement` is unwired from the lab's move/rotate (the export stays for the walk/crowd + its tests).
- `app/dashboard/[eventId]/seating/_components/seating-editor.tsx`: **Save & view.** The 2D→3D switch is now one-tap auto-save-on-switch — the blocking "Save layout first?" modal is retired. A clean switch hops straight through; a dirty one saves then hops when the save settles; on a save failure the layout stays dirty so nothing navigates (the error is surfaced) — a switch never loses work, never blocks. (Removed the now-unused `pendingSwitch3D` state + `Save` icon import.)

**Authoring asymmetry preserved:** create / delete / link / room-size / zones remain the 2D editor's job; the 3D manipulator is move/rotate of existing elements only.

**Deviation surfaced for owner:** the old 3D-only "only a sweetheart table may sit on the stage" rule is dropped for parity — the 2D oracle has no stage constraint (the stage is not a table obstacle in 2D), so honouring "outcomes identical by construction" retires it. Also: the lab already shipped create/delete/link tooling that predates this directive; those are left intact (not ripped back to 2D-only) to avoid destabilising shipped surfaces. Touch already works (pre-existing gesture); this change is validation-only, so no new device scoping.
