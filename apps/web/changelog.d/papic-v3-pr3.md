## 2026-07-19 · feat(papic): v3 pricing recut + free-tier point enforcement (atomic · brief PR-2 code half + PR-3)

One PR, two halves that must never ship apart (owner 2026-07-19 land-together decision): the GBB pricing **recut** (roll→Mini cap remap · wedding-only caps · Free 5→3 · min-paid 5→1) AND the **capture-points enforcement + free-seat provisioning** — so the advertised "3 free cameras" is enforced at the seams the moment it is displayed, never a fake door.

### Half 1 — pricing recut (the paused `claude/papic-v3-pr3` commit)

- **`lib/papic-cameras.ts`** — `CameraCaps` gains `mini`; the `'roll'`/Mini tier now clamps to the **Mini cap** (₱6,000 fallback), not the dormant Ltd cap; `caps.ltd` is reserved for the distinct Ltd tier that ships later. New `isPapicUncapped(eventType)` (`true` for every non-`wedding` type) + `computeCameraQuote(..., { uncapped })`: when uncapped the charge runs to the raw subtotal and `capped` stays `false`. Constant flips: `PAPIC_FREE_CAMERA_COUNT` 5→3, `PAPIC_MIN_PAID_CAMERAS` 5→1.
- **`studio/papic/actions.ts`** — both charge paths (`computeCameraQuote` extras + `computeLimitedQuote` guest-list) read `papic_mini_cap_php` + `event_type` and pass `uncapped` / `MAX_SAFE_INTEGER`; weddings clamp, all other event types bill the raw subtotal.
- **`studio/papic/page.tsx`** — the picker DISPLAY reads the Mini cap for the guest-list Limited tier and uncaps for non-weddings (mirrors the charge path), so the quoted price equals the billed price in every case — closes a would-be overcharge-vs-quote gap for non-wedding events. Stale hardcoded "5 cameras" banner copy now renders from the constants (3 free · 1 min).

### Half 2 — free-tier point enforcement + provisioning (brief PR-3 · the top correctness item)

- **`lib/papic-cameras.ts`** — capture-POINTS model: `papicCaptureCost` (1 photo = 1 pt · 1 five-second clip = 3 pts) + `resolvePointsGate`, the ONE shared fail-posture policy both seams use: **fail-CLOSED on every RPC error EXCEPT function-not-found** (42883/PGRST202 — the seam-cutover carve-out). Budgets are never hardcoded — the PR-1 RPCs resolve them from the admin-editable `papic_tier_config` (free/mini/roll 20 · ltd 70 · unlimited ∞).
- **`provisionFreeCamerasAdmin`** — materializes the Free tier's "always **3 seats / event**" as real `tier='free'` `paparazzi_seats` rows (sku `PAPIC_CAMERA_FREE`, fixed indexes 100..102, idempotent dense top-up), called render-time from the couple's Papic studio page. Before this, no free per-camera seat ever existed, so the "3 free × 20 pts" cap had nothing to bind to — free capture was effectively unlimited.
- **`app/api/upload/route.ts`** (presign seam) — cut from the per-kind `papic_camera_remaining` probe to `papic_camera_points_remaining`: at exhaustion the URL is refused with **409 `camera_points_exhausted`** (no URL ⇒ no orphan R2 bytes); an unidentifiable RPC failure returns 503 `camera_points_unavailable` (fail-CLOSED, retryable).
- **`app/papic/actions.ts`** (record seam, authoritative) — cut from `papic_reserve_camera_capture` to the atomic `papic_reserve_camera_points`; exhaustion → `camera_points_exhausted`, RPC failure → `points_check_failed` (fail-CLOSED; previously this path failed OPEN on thrown errors).
- **`studio/papic/actions.ts`** — `setPapicWindow` re-stamps seat validity from index ≥ 100 (was ≥ 200) so free cameras follow window edits; **`crew/page.tsx`** labels the free range ("Free camera N"); **`papic-seat-capture.tsx`** adds `camera_points_exhausted` to the client cap codes (legacy per-kind codes kept for stale-pairing rollout).
- **`lib/papic-cameras.test.ts`** — 22/22: the original 12 (wedding-clamp + uncapped-non-wedding intact) + 10 new pinning the points cost, the fail-CLOSED/fn-not-found gate policy (exhaustion → 409 · no reservation → no presign), paid tiers unaffected (unlimited NULL-budget passthrough), the legacy PAPIC_SEATS pack un-metered (feature-loss firewall), and the free index range clear of pack + paid ranges.

Code-only — no migration (consumes PR-1's RPCs `20270821110000`/`20270821110100` + PR-2's cap column, all applied to prod). Surfaced consequence: free seats now exist for any event whose couple opens the Papic studio page, so `eventPapicActive` (the add-on prerequisite) is satisfied by the free tier — consistent with the v3 model where add-on gating moves to `eventHasPaidPapicTier` in brief PR-7.
