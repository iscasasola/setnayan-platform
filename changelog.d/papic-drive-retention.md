## 2026-07-22 · feat(papic-storage): PR-4 — Drive-copy retry resilience + guest-capture delivery release + no-Drive hold-and-warn

Final storage hardening (`Papic_Storage_Sustainability_Spec_2026-07-22.md` PR-4 / the design's "Drive resilience + guest-capture release + no-Drive fallback") — a safety net around the free Google-Drive handover and the day-90 full-res drop so a transient failure, or a couple who never connects Drive, can never silently lose an original. Lower-risk than the drop PRs: no new deletion path, it only makes existing paths more robust. Builds on `origin/main` @ #3529 (clip drop).

**1. Autonomous Drive-copy retry (`lib/papic-drive-copy-retry.ts` + `-core.ts`).** `lib/drive-copy.ts` copies each artifact to the couple's Drive at capture/release, but its batch only retries `attempt_count < 5` and ONLY when a new capture/release fires it — so a file that fails 5× (or belongs to an event with no further captures) is stranded forever, and the full-res drop's Guard B then correctly DEFERS its raw forever = a permanent hot R2 leak nothing clears. New cron-free `claimPeriodicJob('papic-drive-copy-retry', DAILY_GAP_MS)` fired from admin `after()` (alongside `maybeRunPapicFullResDrop`, its coupled sibling) re-drives failed copies with **exponential back-off** (30 m · 1 h · 2 h · 4 h … capped 24 h) **up to a hard ceiling** (`DRIVE_COPY_RETRY_CEILING=10`, above the batch's 5); copies stranded PAST the ceiling are **surfaced to admin** (never hot-looped) via `listStrandedDriveCopies()` — rendered as a read-only "Stranded Drive copies" section on the existing `/admin/papic-storage` readout + a structured `console.warn`. `runDriveCopyBatch` gained backward-compatible optional `attemptCap` + `retryDue` params (every other caller unchanged). Non-destructive: only ever ATTEMPTS uploads + bumps `attempt_count`; never deletes.

**2. Guest-capture delivery release (`lib/photo-delivery-release.ts`).** `enqueueRelease` was photo-only (`papic_photos`); it now also lists `papic_guest_captures` (clips + photos, `hidden_at IS NULL`, non-null `r2_object_key`) so a guest-camera original gets the same Release-to-Drive path and is backed up before the guest-inclusive drop. Migration `20270910499209_photo_delivery_artifacts_allow_guest_captures.sql` widens the `photo_delivery_artifacts.source_table` CHECK from `('papic_photos')` to `('papic_photos','papic_guest_captures')` (idempotent, constraint-only, no FK on `source_photo_id`, no RLS change). The `(event_id, source_table, source_photo_id)` dedupe index already keys on `source_table`, so seat + guest rows never collide. `processBatchForEvent` needed no change (source-table-agnostic).

**3. No-Drive retention-lapse safety (hold-and-warn).** Codified: the compressed gallery (display/thumb AVIF) stays on R2-Standard forever — only the full-res original is ever at stake. The existing ~day-76 nudge (`daily-email-jobs.ts runPapicDropWarning`, `WARN_LEAD_DAYS=14`, already "download or connect Drive", already free/no paid upsell) is kept as THE nudge and extended to cover aging **clips** too (gated on `PAPIC_CLIP_DROP_ENABLED` so it doesn't warn prematurely). New **hold-and-warn gate** in the sweep: a `not_connected` event's full-res original is dropped ONLY once the couple was PROVABLY warned (`events.full_res_drop_warned_at` set) AND a lead-time grace (`NO_DRIVE_DROP_WARN_GRACE_DAYS=7`) elapsed — unwarned → HELD (retry next sweep), surfaced via a new `heldNoDriveUnwarned` summary counter + `console.warn`. This subsumes the design's separate "Keep-Full-Res lapse" mechanism: a lapsed `HIGH_RES_ARCHIVE` event (warn stamp null while the SKU was active, was skipped by the warn job) is HELD on lapse until the now-eligible nudge lands. The retired paid "Keep Full-Res" upsell (owner 2026-07-17, #3523) is NOT reintroduced — the fallback is download-in-window / connect-Drive / else hold-and-warn, all free. Gate is behind `PAPIC_DROP_REQUIRE_WARN` (**safe-default ON**: it can only ever PREVENT a deletion, so on-by-default is the safe direction; prod has only excluded sample photos today so it changes nothing live). Clip drop is NOT re-armed (still `PAPIC_CLIP_DROP_ENABLED`, off).
