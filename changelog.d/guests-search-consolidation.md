## 2026-07-13 · refactor(guests): one always-visible search — retire the duplicate + the Add|Find toggle

The desktop Guests (Living Roster) page rendered the SAME `live-search.tsx` component **twice** — once behind the CaptureBar's `[Add | Find]` toggle (P2, 2026-07-11) and once in the Toolbar below — both writing `?q=` live. Because the CaptureBar defaults to **Add**, the Toolbar box was the only search input in the DOM on landing; toggling Find stacked a second identical box on top (the redundancy the owner spotted). A 4-lens design council ([`Guests_Search_Consolidation_Council_Verdict_2026-07-13.md`](../../Documents/Claude/Projects/Setnayan/Guests_Search_Consolidation_Council_Verdict_2026-07-13.md)) unanimously flagged that deleting the duplicate but leaving search "only behind Find + ⌘K" ships an a11y/discoverability regression (zero `input[type=search]` on landing). Owner-approved the refined resolution 2026-07-13.

- **CaptureBar → Add-only** (`capture-bar.tsx`): the `[Add | Find]` toggle, `addMode`/`switchTo`, and the Find-mode `<LiveSearch>` are removed. It is now purely the capture-first guest parser + "More ways" overflow. The ⌘K/Ctrl-K listener moved out to the search island.
- **One always-visible search** (`_components/guests-search.tsx`, new): wraps `live-search.tsx` VERBATIM in a `role="search"` landmark + a visible ⌘K hint, and owns the ⌘K shortcut. Rendered in a new **query row** at the head of the `SummaryFacetBar` (under the meters, above the facet-pill rows) — so search is visible on landing without toggling, mirroring the mobile carousel's persistent compose-row.
- **Sort is now instant** (`_components/sort-select.tsx`, new): replaces the Toolbar's native `<form method="get">` + **Apply** submit (a full-page reload with 7 hidden param-preserving inputs) with a client island that merge-writes `?sort=` via `router.replace` — reading `searchParams` inside the handler so a concurrent filter click is never clobbered (same contract as `LiveSearch`). `?sort=group` still drives server-side group bucketing. Sort now behaves like the instant filter pills + search; the last full-reload control on the surface is gone.
- **Toolbar deleted** (`page.tsx`): the `Toolbar` component + its call site are removed; Search + Sort + the List/Mind-map switch fold into the facet-bar query row. Desktop chrome drops from **three blocks to two**. The `q`/`sort`/`sortOptions`/`gview` props thread into `SummaryFacetBar`.

Mobile is untouched (all edits are inside the desktop `hidden lg:block` wrapper; `mobile-guest-carousel.tsx` keeps its own compose-row). The full URL param contract (`q/rsvp/view/group/team/tag/sort/gview`) is preserved — every control stays a URL writer (SSR + shareable). No migration. Verified: typecheck + eslint + guest-legibility clean; SSR render of the real components shows exactly one `type=search` input (was two), a `role="search"` landmark, no `aria-pressed` toggle, and the instant `Sort guests` select (7 options, no `>Apply<`).

SPEC IMPACT: **Supersedes the 2026-07-11 P2 "CaptureBar = single doorway for both Add and Find" owner-lock** — Find is no longer a mode-peer of Add; search is an always-visible lens in the facet bar. Owner-approved 2026-07-13. Logged in `DECISION_LOG.md`; rationale in `Guests_Search_Consolidation_Council_Verdict_2026-07-13.md` (spec corpus).
