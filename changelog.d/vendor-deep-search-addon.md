## 2026-07-22 · feat(vendor-billing): Deep Search — a sellable, per-use ₱500 metered add-on (1 free/28d on Pro+) that runs the web-research dossier on the vendor's OWN business

Owner-locked 2026-07-22: **Deep Search** — a **₱500-per-search** metered add-on that lets a **verified, paid-tier** vendor run the web-research deep search on their **OWN business** and review a "what we learned" result to auto-fill their Shop profile. It is the **vendor-run** version of the admin verification deep search and **REUSES the same engine + store** (`apps/web/lib/vendor-deep-search.ts` · `runDeepSearchOrLite` — the Haiku AI dossier when `ANTHROPIC_API_KEY` is set, else the ₱0 keyless Lite pass — and the `vendor_web_dossiers` table). No second web-gatherer was built. The engine's implicit master switch is unchanged: **no key → the free Lite pass runs** (kept as-is, not flipped).

- **Eligibility:** all PAID tiers (**Solo / Pro / Enterprise / Custom**), **verified only** — NOT the free / verified-only tier.
- **Free allowance:** **Pro / Enterprise / Custom get 1 FREE search per 28-day cycle, then ₱500 each. Solo pays ₱500 EVERY time** (no free allowance).

**Migration** `supabase/migrations/20270907924171_vendor_deep_search_sku.sql` — (1) extends the `vendor_billing_catalog` `offering_type` + `vendor_billing_shape` CHECKs with a new **`vendor_addon_metered`** value (same drop/recreate pattern as `20270907628470`; includes EVERY prior value incl. `vendor_addon_per_event` + `vendor_addon_recurring`); (2) INSERTs the admin-managed **`vendor_deep_search`** SKU (₱500, `vendor_addon_metered`, display_order 84, `ON CONFLICT DO UPDATE` that never stomps an admin price edit); (3) creates the usage-log table **`vendor_deep_search_uses`** (`vendor_profile_id` · `used_at` · `was_free BOOL` · nullable `order_id` · nullable `dossier_id → vendor_web_dossiers.id`) with **RLS at CREATE TABLE** (vendor-own read via `current_vendor_profile_ids()`; admin all; no vendor-write policy — a vendor can never mint its own free-allowance row), a `(vendor_profile_id, used_at DESC)` index, and a **partial `UNIQUE(order_id) WHERE order_id IS NOT NULL`** so a re-run of the approval hook can't double-count a paid ₱500 order. Idempotent. The ₱0 free search is a price RESOLVER (not a ₱0 catalog row — the `price_php > 0` CHECK forbids it).

**How the 1-free-per-cycle allowance is counted / cycle-start definition** — `apps/web/lib/vendor-deep-search-addon.ts` (all PURE, unit-tested): `deepSearchCycleStartMs(tierExpiresAt, now)` anchors the 28-day allowance cadence to the vendor's subscription window — the current cycle start is the most recent 28-day boundary walking **backward from `vendor_profiles.tier_expires_at`** (so it resets every 28 days in phase with billing, identically for monthly and annual subscribers); **fallback** (documented) when `tier_expires_at` is missing/invalid = a **rolling 28-day window ending now**. `countDeepSearchUsesSince` COUNTs the vendor's `vendor_deep_search_uses` rows since that start (soft-fails to a safe HIGH count so a broken read never hands out a free search it can't record). `resolveDeepSearchPricePhp({ tier, usesThisCycle, cyclePricePhp })` → **Pro/Ent/Custom → ₱0 when 0 uses this cycle else ₱500; Solo → always ₱500** (catalog value, ₱500 fallback). `deepSearchEligibility({ tier, verification })` gates tier→verification (free/verified → `tier_too_low`; unverified → `unverified`). Tests in `vendor-deep-search-addon.test.ts` (18 cases incl. **Solo-always-₱500**, **Pro-first-free-then-₱500**, Enterprise/Custom parity, **free-tier ineligible**, cycle boundary/annual/fallback math, and an integration check that a use inside vs. before the cycle flips the price).

**Run seam (the real path — the search actually runs + shows output)** `apps/web/lib/vendor-deep-search-run.ts` — server-only `runAndRecordVendorDeepSearch(admin, {...})` opens a `running` dossier row, calls `runDeepSearchOrLite`, flips it to `complete`/`failed`, and on **success only** writes the `vendor_deep_search_uses` row (a failed run never burns the allowance). Shared `buildVendorDeepSearchInputs` snapshots the vendor's own profile facts (name · website · social from gallery links · city · services) so the run action + the approval hook build identical inputs.

**Run action + gate** `apps/web/app/vendor-dashboard/deep-search/actions.ts` — `runVendorDeepSearch` rejects free/verified + unverified **before pricing** (the orders spine has no other gate for a vendor add-on), COUNTs uses since the cycle start (admin-read for authority), and re-reads the authoritative ₱500 + `is_active` server-side. **FREE search (resolved ₱0) → runs NOW**, records `was_free=true`, returns the dossier inline. **PAID search (₱500) → apply-then-pay** `orders`+`payments` (lands in `/admin/payments`).

The vendor genuinely gets their dossier output either way.
