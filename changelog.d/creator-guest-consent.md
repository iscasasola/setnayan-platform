## 2026-07-17 · fix(privacy): creator teaser honors guest showcase-consent + geo-strip + vendor-relationship gate (RA 10173)

The creator "Adventure Chapter" publish + owned-music teaser path skipped consent gates the couple-RECAP path already enforces. This applies the platform's EXISTING consent model to that path — no new policy, no publish-time attestation (owner dropped that as over-strict for a self-asserted platform). Three consent-parity fixes, each reusing the recap path's gate (`app/[slug]/_components/editorial/data.ts`):

**GAP-1 (acute) — teaser guest-consent.** `buildChapterTeaserPlan` (`lib/creator-teaser.ts`) pulled up to 8 photos through `fetchPapicGallery`, which filters only `nsfw_blocked`/`hidden_at` — the consent columns were SELECTed but never filtered, and for `kind='photo'` guest captures they were left unset (they "ride only on clips" in that reader). So an un-consented guest photo could reach a PUBLIC, shareable teaser. Fix: a new teaser-scoped read `fetchTeaserFrames` (`lib/papic-gallery.ts`) that mirrors the recap's public gates EXACTLY, per capture table:
- SEAT captures (`papic_photos`): excluded when moderation-withheld (`moderation_state IN (nsfw_blocked, consent_withheld, faceblock_withheld)`) or couple-hidden — identical `.not('moderation_state','in',…)` to the recap; `'unscreened'` fails open, same as the recap gallery read.
- GUEST captures (`papic_guest_captures`): the DOUBLE consent gate the Alaala public showcase enforces — `consent_to_public = TRUE` (the guest opted in at capture) AND `couple_approved_for_showcase = TRUE` (the couple picked it) AND `hidden_at IS NULL`. This table carries no NSFW state; the two gates ARE its public gate. Confirmed both columns apply to guest PHOTOS (migrations `20270215631984` + `20270216612756`; the recap gates guest photos on both). Zero cleared frames → teaser renders `canRender:false` gracefully, NEVER falls back to unapproved guest media. `fetchPapicGallery` is left untouched for its couple-dashboard callers.

  Note on scope: `couple_approved_for_showcase` on the SEAT table (`papic_photos`) has NO producer for photos ("rides only on clips"), and the recap does NOT gate seat photos on it — it gates them on `moderation_state`. So this fix does NOT extend that column to seat photos (that would be a schema/consent-flow change = owner+counsel call); it mirrors the recap's per-table model instead. The acute risk (un-consented GUEST faces) is closed via the guest-capture double gate.

**GAP-4 — teaser geo.** Teaser frame URLs now resolve ONLY the metadata-stripped `display_r2_key`/`thumb_r2_key` derivative (sharp AVIF re-encode drops all EXIF/GPS by default — `lib/papic-derivatives.ts`), NEVER the geo-bearing `r2_object_key` original. A frame with no such derivative is SKIPPED (no fall-through). RA 10173 · CLAUDE.md "geo stripped on outbound shares."

**GAP-3 — vendor attribution.** `resolveShoppableVendors` (`lib/creator-public.ts`) rendered a public shoppable/bookable vendor card for ANY creator-typed `substrate.vendor_ids`, manufacturing a Setnayan-looking "Book this vendor" affordance for a vendor with no relationship. Fix: a card links to `/v/[slug]` + shows the Book CTA ONLY when a real relationship exists — an ACCEPTED `vendor_creator_offers` collab with this chapter's creator, OR a genuine booking tying the vendor to the chapter's event (`event_vendors.linked_vendor_profile_id`, keyed on `substrate.papic_gallery_id`). Otherwise the vendor renders as PLAIN TEXT (name only, no link, no Book CTA). Fail-closed: any relationship read error → plain text. `app/u/[userSlug]/c/[chapterId]/page.tsx` passes the creator/event context and branches card-vs-text.

Verify: typecheck + lint clean; `migration:check` green (no schema change — read-path only). Traced: event with 0 showcase-approved guest photos → teaser has no guest faces; unrelated `vendor_id` → plain text not a card.
