## 2026-07-03 · feat(vendor): wizard parity — pricing bases, lists, media + parent-cap & faith-vocab unification

The guided "create a service" wizard (`/vendor-dashboard/services/new/[category]`) was a generation behind the inline My Shop editors — a vendor onboarding through it got a flat-price-only card and its raw `crew_meal_required` checkbox could contradict the card ("crew meal not included" when it's included). Brought to full parity by REUSING the shipped inline components:

- **Wizard UI** (`service-wizard.tsx`): the flat starting-price input is replaced by `PricingBasisEditor` (fixed / per-guest / per-hour) with `PriceBracketsEditor` gated to the Fixed basis via `fixedExtra`; `IncludedFlags` (crew meal + transport, defaults not-included) replaces the old bare "Crew meal required" checkbox; a new "Value & media" step mounts `InclusionsEditor` + `DiscountsEditor` + `ShowcaseMediaFields`; `ServiceCardLivePreview` is mounted at the top of the form (the wizard is ONE `<form>` with hidden step sections, so the preview honestly mirrors the exact card across all steps). The advanced-details duplicate base_pax / added_pax fields were removed (they live in the Fixed branch now).
- **`commitVendorService`** now mirrors the inline actions: `parsePricingFields` (+ lowest-bracket "from ₱X" anchor rule), `parseShowcaseMedia`, `parseDiscountRows`/`parseInclusionRows`/`parseBracketRows` (brackets Fixed-only), `crew_meal_included`/`transport_included`/`transport_flat_fee_php` with `crew_meal_required = NOT crew_meal_included` (0007 budget bridge), and passes real `p_discounts`/`p_brackets`/`p_inclusions` to the `save_vendor_service` RPC. The legacy single-discount scalar path (`discount_value` + unit heuristic) is removed — the wizard submits the same multi-discount arrays as the inline form. Showcase photos get the QR-in-media guard (all-new on create; fresh-only on edit). Publish gates, cover QR guard, NSFW hash-scan `after()`, claim-flow redirect, and revalidates preserved.
- **Parent-category cap unification** (`createVendorService` + `commitVendorService`): the cap now counts the UNION of existing services' category-parents ∪ the vendor's `vendor_coverages` parents (canonical_service → tier-1 folder via `getCoverageTaxonomy()`; tier-1 `service_categories.id` values ARE the `TILE_PARENT` folder vocabulary). Coverage is becoming the source of truth for what a vendor offers. Fail-soft: a taxonomy/coverage read error degrades to the legacy services-only count.
- **Faith validation source** (`coverage-actions.ts` + new `lib/faith-vocab-db.ts`): `parseFaiths` now validates against ACTIVE `faith_vocab.faith_key` rows (admin-managed, zero-deploy) via a React-`cache()`d `getActiveFaithKeys()` helper, falling back to the static `FAITH_REGISTRY` faithCol set on any read error/empty result — the same fail-soft contract as `lib/event-types-db.ts`.

No migration — the 9-arg `save_vendor_service` RPC and all columns already live in prod (migration 20270502342558).

SPEC IMPACT: None — parity + consistency fixes; no new product surface, pricing, or schema.
