## 2026-07-21 · feat(vendor-onboarding): the shop logo moves from required-to-REGISTER to required-before-VERIFICATION

Owner decision 4 (2026-07-21, verbatim): _"shop logo is only required before verification. starting your shop can start as name, next is completing the profile, then verification."_

**What was wrong.** `/open-shop` hard-refused to create a shop without a logo — server-side at `app/open-shop/actions.ts` (a redirect that fired BEFORE the `vendor_profiles` row was even found or inserted, so a logo-less vendor could not finish onboarding at all) and client-side in `open-shop-wizard.tsx`'s `next()`, which blocked the step 1 → step 2 advance. A vendor who wanted to start with just a name had no path in.

**The change is narrow because the three-stage model the owner described was already ~90% built.** Stage 1 ("start as a name") is the `business_name` routing flag at `app/open-shop/page.tsx`; stage 2 ("completing the profile") is the 8-item `businessProfileChecklist` on My Shop; stage 3 is the Get-verified section. And _"logo required before verification"_ was **already shipped** — `lib/vendor-verification.ts` `verificationSubmitMissing()` refuses submit unless `businessProfileChecklist().complete`, and the logo is item #1 of that checklist. So the authorised change was to stop demanding the logo at registration, and to close the one hole that made that unsafe.

- **`lib/open-shop-validation.ts`** — new shared `OPEN_SHOP_LOGO_REQUIRED: boolean = false`. This module exists specifically because the wizard's client gate and the server action once drifted about what "valid" means; required-ness is now ONE value both layers import, rather than two independently-deleted `if` blocks that can be independently restored. Typed `boolean` (not the literal `false`) so both call sites stay real branches and flipping this single value restores the old rule. `OPEN_SHOP_ERRORS.logo` is kept — the sentence is still correct if the flag is flipped back.
- **`app/open-shop/actions.ts`** — the logo redirect is now `if (OPEN_SHOP_LOGO_REQUIRED && !logoUrl)`. **Clobber fix in the same commit:** `logo_url` was written unconditionally into the update patch. That was safe only because the removed guard made a null logo unreachable; without it, re-running the wizard on a shop that already had a logo (the `mode='complete'` path) would have NULLed it, breaking the module's own documented "existing non-empty values are never clobbered by blanks" contract. Now guarded exactly like `location_city`.
- **`app/open-shop/_components/open-shop-wizard.tsx`** — same shared flag drives the `next()` gate, the label ("Shop logo · optional for now" instead of a required asterisk), and the helper copy. That copy was also **factually wrong** — it said "up to 2 MB" while the `<FileUpload>` two lines above has been `maxSizeMB={10}` and accepts HEIC/HEIF since #3457. It now states the real limit and carries the obligation forward: _"You can add it later from My Shop — but you'll need it to publish your shop and to get verified."_
- **`app/vendor-dashboard/verify/actions.ts` — closes a pre-existing hole that this change would otherwise have made live.** There are two submit-for-review paths. The My Shop inline twin (`shop/inline-docs-actions.ts:submitInlineForReview`) has always enforced `verificationSubmitMissing`, which requires a complete profile. `/vendor-dashboard/verify`'s `submitApplication()` checked only the document count — so it could flip `verification_state → 'pending_review'` with a NULL `logo_url`. Harmless while no logo-less vendor could exist; load-bearing the moment they can. Both routes now read the one shared gate. Only the PROFILE reason is taken from it: this page's document rule is deliberately the launch-soft `VENDOR_DOC_SLOTS` count, and folding the two together would tighten a second thing this change is not authorised to touch.
- **`app/vendor-dashboard/shop/_components/profile-checklist-editor.tsx`** — one line of copy. The obligation is surfaced on the surfaces that already exist (no new widget): the sub-100% ring, the Profile tile's "N fields left", and a checklist row literally labelled **"Logo"**. The sub-line previously said only that completing the profile is what lets couples contact you; it now also names the second consequence — it is what unlocks getting verified.

**The requirement moved; it did not disappear.** A logo-less vendor still cannot **publish** (`app/vendor-dashboard/actions.ts` forces `is_published = false` and names the missing Logo) and still cannot **get verified** (both submit paths). Deliberately untouched: the publish gate (kept strict — a listed shop still shows couples a logo), the wizard's 2-step shape, and the `logo` checklist item, which is the single thing making both downstream gates true.

Tests — new `lib/open-shop-logo-gate.test.ts` (5 cases, `node:test`): the flag is off; a logo-less profile is 7/8 with `missing === ['Logo']` (the obligation is named, not silent); the logo is in the label set the publish gate shares; a logo-less vendor with **every required document uploaded** still gets exactly `['Finish your business profile']` from the verification gate; and adding the logo is the only thing that clears it. The suite is written so that dropping the logo from the checklist to "make onboarding easier" fails loudly — that one edit would unlock publish AND verification at once.

Verified: `tsc --noEmit` exit 0 · `next lint` exit 0, zero findings on the touched files · `tsx --test "lib/**/*.test.ts"` = 2450/2450 pass (includes the pre-existing `vendor-profile-completeness.test.ts`, which pins the logo as a checklist item and stays green).
