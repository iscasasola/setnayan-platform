## 2026-07-11 · feat(vendors): surface completion handshake in the relationship shell + code-split heavy tab components

Two flag-gated shell improvements behind `NEXT_PUBLIC_RELATIONSHIP_WORKSPACE_ENABLED` (default OFF). Both changes touch ONLY the flag-ON render path — the flag-OFF long-scroll / CardTabs pages stay byte-identical.

**PR13 — completion handshake, now inside the tabbed shell (both sides).** The service-completion handshake (`vendorMarkServiceComplete` → `coupleConfirmReceived` / `coupleReportNonDelivery` → the review-request emit) was built server-side but only reachable OUTSIDE the shell tabs. Now surfaced in each side's Details tab, reusing the existing actions + `reviewState` (no handshake / review-emit logic reimplemented):

- **Couple** (`dashboard/[eventId]/vendors/[vendorId]/workspace/page.tsx`): new `coupleCompletionSection` in the **Details** tab. `awaiting_confirm` → the "Yes, I got everything" (`coupleConfirmReceived`) + "Something's missing" (`coupleReportNonDelivery`) pair (previously only on the standalone `/review` page); `reviewable` → a "Leave a review" nudge, suppressed when the marketplace-info section already shows the link (legacy status delivered/complete) so it never doubles up; `disputed` → the on-hold note; `awaiting_vendor` → nothing. Completion fields + `events.event_date` are fetched in a flag-ON-only read (couple RLS), so flag-OFF adds zero queries. Connected (marketplace) vendors only — an off-platform/manual vendor has no account to mark complete and no profile to review, so it degrades to nothing.
- **Vendor** (`vendor-dashboard/clients/[eventId]/page.tsx`): the "Mark service complete" Card was buried mid-brief inside `OverviewTab`. Extracted verbatim into a module-level `VendorCompletionCard` (single source of truth), rendered as a first-class node at the top of the **Details** tab on flag ON — a true two-sided mirror of the couple. `OverviewTab` takes a new `hideCompletion` prop (`= relationshipShellEnabled`); flag OFF passes `false` so the card still renders inline exactly where it did (identical markup), flag ON passes `true` so it isn't shown twice.

**Light lazy — code-split the WebRTC Call launcher.** New `app/_components/thread-call-launcher-lazy.tsx` client loader wraps `ThreadCallLauncher` in `next/dynamic({ ssr: false, loading })` (seating-lab / veil-reveal loader pattern). Both workspace pages import the lazy loader for the Call tab, so the WebRTC bundle (`thread-call-room` + `lib/call-webrtc`) no longer ships in the initial page JS — it loads only when the Call tab actually mounts. Only the leaf is client-lazy; the surrounding server-rendered tab content is untouched. Left the chat stream (`ChatMessageStream`) as a static import on purpose: it's the landing tab and SSRs its first message batch (no-flash), so lazy-loading it would regress the primary path, not improve it.
