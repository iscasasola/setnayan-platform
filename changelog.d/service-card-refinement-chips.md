## 2026-07-03 · feat(vendor): inline refinement chips on the fast service-card form

Closes the last v20 service-card gap: a vendor can now tag a leaf's **refinements** (the iteration-0044 category-specific facets — `shooting_style`, `cuisine_specialties`, `music_genres`, …) right on the create/edit card, instead of detouring to the separate `/vendor-dashboard/attributes` tool. Refinements power marketplace filtering, so wiring them into the fast flow lifts fill rates where vendors actually work. Surfaces the EXISTING substrate — no schema change.

- **New `RefinementsEditor`** (`services/_components/refinements-editor.tsx`): a sibling `<form>` (mirroring the "Comes with" / payment-schedule pattern) that renders only the **chip-shaped** category-specific fields (`multi_select` / `enum` / `boolean`) via the same `AttributeFieldRenderer` the full tool uses — identical look + `field__<key>` names, pre-filled from the vendor's saved payload. Heavier fields (int / free-text / URL lists) and the shared faith/dietary/region groups stay in the full tool, reachable via a "More details →" link. Wired into BOTH the edit panel (per service, when its category has chip refinements) and the inline "Add: <category>" form.
- **New `saveServiceRefinements` action** (`services/refinement-actions.ts`): **MERGE** semantics — overlays only the submitted, schema-validated, chip-typed keys onto the existing `vendor_service_attributes` payload and preserves everything else, so a quick chip edit can never wipe fields set in the full tool. Re-checks each key is chip-shaped before it can write (hardened against crafted POSTs). Recomputes `meets_visibility_minimum` + `completeness_score` over the full merged payload so the listing-ready badge stays truthful. Keyed by `canonical_service` (= `vendor_services.category`); returns to the vendor's surface via `servicesReturnBase()`.
- **Parse/score helpers hoisted to `lib/vendor-service-attributes.ts`** (`parseAttributeFieldValue`, `checkAttributeConditionalRequired`, `isAttributeFieldFilled`, `computeAttributeCompleteness`, `ATTRIBUTE_FIELD_NAME_PREFIX`, URL regex) so the full attributes tool and the inline chips share ONE source of parse truth — no drift. `attributes/actions.ts` now imports them (behaviour unchanged). Added `fetchCategoryChipRefinements` / `fetchCategoryChipRefinementsMany` (read `category_specific_attributes`, keep only chip types).

No migration — reuses `vendor_service_attributes` + `canonical_service_schemas` (iteration 0044, already in prod). Refinements are category-level (one payload per leaf), and the editor copy says so.

SPEC IMPACT: None — surfaces the existing 0044 refinements substrate inline on the fast card; no new schema, pricing, or locked-decision change. (Tracked as the remaining v20 gap in the service-card prototype design.)
