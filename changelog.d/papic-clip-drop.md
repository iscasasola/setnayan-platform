## 2026-07-22 · fix(papic): storage PR-2 clip drop was INERT — wire isClipRow to the real sweep Item + close the test-masking gap

The clip branch shipped switched OFF, but it was also **non-functional by construction**: even with `PAPIC_CLIP_DROP_ENABLED='true'` it would have dropped **zero** clips — and, worse, its irreversible-deletion guard chain (HEAD custody → `r2Delete`) had **never actually executed against the real sweep data shape**. `clipEligibleForDrop` gates on `isClipRow(row)` FIRST, and `isClipRow` reads ONLY `photo_type`/`media_type` — but the clip SELECTs fetched neither column and the clip `.map()` set `kind:'clip'` without either type field, so every real clip Item had `isClipRow=false → clipEligibleForDrop=false → clipsDropped` stuck at 0 and the delete path was dead code. The +21 unit tests passed only because their fixtures hand-built a `ClipDropCandidate` with `media_type:'clip'` pre-set; the DB test ran raw SQL, not the sweep wiring — so both went green over a dead production path.

- **`lib/papic-fullres-drop-core.ts`** — extracted the four sweep row→Item mappers (`seatPhotoItem`/`guestPhotoItem`/`seatClipItem`/`guestClipItem`) + the `PapicDropItem` type here as PURE code, so the sweep AND its regression test materialise candidate Items through the SAME builders (a hand-built fixture is exactly what masked the bug). The clip builders CARRY the type column (`photo_type`/`media_type`) so `isClipRow()` is genuinely true for a real sweep clip Item. New pure `sameResolvedObject(a, b)` (defense-in-depth): two DIFFERENT key strings can resolve to one R2 object (legacy `key` vs `r2://setnayan-media/key`), so resolved-level distinctness is asserted before the delete — fail closed on collision.
- **`lib/papic-fullres-drop.ts`** — clip SELECTs now fetch `photo_type` (seat) / `media_type` (guest); the four `.map()` literals now call the shared builders; the clip custody gate asserts `sameResolvedObject(clip_web_r2_key, r2_object_key)` is false before HEAD/delete. A real sweep-built clip Item now yields `isClipRow=true` and reaches the guard chain.
- **`lib/papic-fullres-drop-wiring.test.ts` (new)** — the test that was missing. It builds the Item EXACTLY as the sweep's `.map()` does (through the production `seatClipItem`/`guestClipItem` mappers, from a raw DB-row shape identical to the clip SELECT — NOT a pre-set `ClipDropCandidate`) and runs it through `clipEligibleForDrop` + the custody gate. Proves: (a) an eligible real clip Item DROPS; (b) each attack still fails closed via the real Item shape (null web key, web==poster/display/raw, bytes<1024, null HEAD, size≠bytes, non-video, within-7d-grace, young, already-dropped, sample/ key, resolved-collision); (c) the exact original bug — a row whose SELECT forgot the type column → INERT Item — is asserted; (d) only the raw is targeted (web copy + poster resolve elsewhere). Mutation-checked: reverting the mapper's type-carry makes the crux "eligible clip DROPS" cases FAIL (4/9), so the destructive path can no longer regress to inert or unguarded while green.

No behaviour change while the flag is unset (clips still never queried); no migration, no `service_code` rename, no `is_active` flip, no capture/metering/face change.

SPEC IMPACT: None — the clip drop remains gated behind `PAPIC_CLIP_DROP_ENABLED` (OFF by default) exactly as the storage spec documents; this only makes the already-specified behaviour actually reachable and adds the production-path test.

## 2026-07-22 · feat(papic): storage PR-2 — extend the full-res drop to CLIPS (the cost win, gated OFF)

Builds on storage PR-1 (#3521 — `clip_web_r2_key`/`clip_web_bytes` columns, web-copy at capture, `resolvePlayRef` coverage). A raw clip's ~2× (10s) bytes now become reclaimable at day 90 once a durable, verified playable web copy exists — the raw is dropped, the poster still and the web copy survive forever. This IRREVERSIBLY DELETES bytes, so the whole change is guard-first and ships **switched OFF**.

- **`lib/papic-fullres-drop-core.ts`** — new pure `clipEligibleForDrop(row, opts)` (column-level) + `clipWebCopyCustodyOk(head, expectedBytes, opts)` (object-level), plus `CLIP_WEB_DROP_GRACE_DAYS = 7` / `CLIP_WEB_MIN_BYTES = 1024`. A clip's raw is droppable ONLY when: not already dropped · is a clip (photos keep `isEligibleForDrop` — never cross the streams) · `clip_web_r2_key` non-null · that web key is DISTINCT from `poster_r2_key`/`display_r2_key`/`r2_object_key` (poster-trap, via `clipWebKeyDistinct`) · `clip_web_bytes >= 1024` · raw key present + not `sample/` · past the 90-day fuse. `clip_web_r2_key IS NULL ⇒ INELIGIBLE` is what keeps every existing / un-backfilled clip safe before PR-3's backfill.
- **`lib/r2.ts`** — new `r2Head()` (HeadObjectCommand → `{size, contentType, lastModified}`), fail-safe (`null` on missing/403/network). The custody check requires the live web object to EXIST, `size === clip_web_bytes` (a truncated PUT mismatches), `content-type video/*`, AND be `>= 7 days` old by R2 `LastModified` — the **fresh-grace** window, measured on the real object (no new column, no capture-code change) so a bad/late backfill transcode can never convert straight to raw loss.
- **`lib/papic-fullres-drop.ts`** — clip candidates are queried per table (`photo_type/media_type='clip'` + `clip_web_r2_key IS NOT NULL` + age + not-dropped) and merged with a `kind` discriminator; each clip runs `clipEligibleForDrop` → Keep-Full-Res skip → **Guard B `isDriveDeferred`** (clip keys included in the per-event `confirmedKeys` batch; `drive_copy_artifacts` already holds clip rows) → **HEAD custody + grace** → delete ONLY `resolveOriginalRef(r2_object_key)` (the raw) and stamp `full_res_dropped_at` (poster + web copy untouched; `resolvePlayRef` then serves the web copy). New summary fields `clipDropEnabled`, `clipsDropped`, `clipWebUnverified` + a kept-clip observability warn.
- **Go-live gate:** new `clipDropEnabled()` = `PAPIC_CLIP_DROP_ENABLED === 'true'` (OFF by default). Migrations auto-apply on merge, so the gate is an env flag, not "hold the migration": with the flag unset the sweep never even QUERIES clips and behaves exactly as today (photos only). The master `PAPIC_FULLRES_DROP_ENABLED='false'` kill-switch still forces dry-run for everything, clips included.
- **Presign fleet re-audit:** confirmed live clip playback (`papic-gallery`, `alaala-orb`) already routes through `resolvePlayRef` (excludes the dropped raw); `guest-live-gallery` excludes clips and uses still derivatives. No surface presigns a raw clip after `full_res_dropped_at`.
- **Tests (this deletes data — the guards are proven):** `papic-fullres-drop-core.test.ts` (+21 cases) covers NULL web key, web==poster/raw, below byte floor, young/dropped/sample, and every custody failure (null HEAD, within-grace, size-mismatch, non-video, no LastModified) + the survivors invariant (delete target distinct from poster/web; `resolvePlayRef`→web, `resolveStillRef`→poster after the stamp). New `tests/db/papic-clip-drop.db.test.ts` proves, on the replayed prod schema, that the candidate query excludes NULL-web-copy clips and the drop UPDATE touches ONLY `full_res_dropped_at` (poster/display/web key/raw key intact) on both capture tables.

No new migration (PR-1's columns suffice). No `service_code`/`tier_code` rename. Capture/metering/face paths untouched.

SPEC IMPACT: storage — clips become droppable at day 90 once a verified web copy exists, reclaiming the raw-clip bytes (the ~95% forever-affordability win per `Papic_Storage_Sustainability_Spec_2026-07-22.md`).
