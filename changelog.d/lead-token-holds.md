## 2026-07-12 · feat(anti-fraud): Phase B — token hold-and-release on inquiry accept

Second slice of fake-inquiry protection (corpus: `Vendor_Fake_Inquiry_Protection_Build_Plan_2026-07-11.md`). Turns the vendor's accept charge from a **burn** into a **hold**: the token is reserved at accept and only truly consumed when the couple genuinely replies; a ghost auto-releases it. Fakes never reply → their hold is returned → the vendor never pays for a fake. Owner-approved 2026-07-12 ("build it now, flag-gated").

- **`supabase/migrations/20270726988829_lead_token_holds.sql`** — `lead_token_holds` ledger (one per vendor,event; RLS: vendor-team read / admin) + four SECURITY DEFINER RPCs: `unlock_vendor_event_hold` (a PARALLEL accept path mirroring every gate/error of the live `unlock_vendor_event` but HOLDING instead of burning, with a reservation check `available − held ≥ tokens` that reads the same balance sources the burn draws from so a founder holding only earned vouchers is never falsely blocked), `consume_lead_token_hold(_for)` (debits via the existing `consume_*` on a real reply; idempotent), `release_lead_token_hold`, and `sweep_ghosted_lead_holds`. **The live `unlock_vendor_event` is UNTOUCHED.** Strategy: no debit until consume, so `available = balance − SUM(held)` and release needs no fragile reversal of FIFO earned-voucher consumption.
- **`apps/web/lib/lead-token-holds.ts`** — flag `NEXT_PUBLIC_LEAD_TOKEN_HOLD_ENABLED` (default OFF) + thin RPC wrappers (`as never` per the not-yet-in-generated-types convention).
- **`apps/web/lib/chat-actions.ts`** — `acceptInquiry` branches to the hold RPC when the flag is on; both raise the same TIER/LIMIT/BALANCE errors so the existing handling is unchanged. Flag OFF → the live burn RPC, byte-identical.
- **`apps/web/lib/chat-send.ts`** — a genuine couple reply on an `accepted` thread fires `consumeLeadHoldOnCoupleReply` off the request path (`after()`), idempotent, dormant unless the flag is on.
- **`apps/web/app/api/cron/lead-hold-sweep/route.ts`** + `vercel.json` — daily cron releases holds still 'held' past the 7-day ghost window (CRON_SECRET-gated, fail-closed; mirrors the retention-sweep route). Safe to run even flag-off (no holds → 0 released).

Merging changes nothing until the owner (a) applies the migration (`supabase db push`) and (b) sets `NEXT_PUBLIC_LEAD_TOKEN_HOLD_ENABLED=true`. Known bounded v1 behaviors (documented, acceptable per the presumption-of-a-real-couple bias): a couple that ghosts-then-returns after release, or a vendor who drains their balance between hold and reply, yields an un-charged real lead — rare, vendor-favorable, and swept-safe.
