## 2026-07-24 · feat(vendor): unique government registration number (anti-farm identity)

A vendor's launch perks (e.g. a "first N bookings free" window) were farmable because NOTHING about a vendor's identity was unique: `business_name` / `contact_email` / `contact_phone` are all free-text non-unique, and the DTI / BIR-2303 / Mayor's-Permit "documents" lived only as opaque R2 file refs inside `vendor_verification_applications.doc_uploads` JSONB — with no extracted, comparable number. One person could open a second account, re-upload the same papers, and reset the perk clock. This makes the vendor's government registration number a UNIQUE identity so a duplicate account can't slip through unnoticed.

- **Migration `20270925937630_vendor_registration_number_unique.sql`** (own concern, RLS unchanged) adds to `vendor_profiles`: `registration_number_raw` (what the vendor typed, for display + admin review), `registration_number_normalized` (UPPERCASE + non-alphanumerics stripped — the canonical comparison key), `registration_number_needs_review` (collision flag), and `registration_number_submitted_at`. The anti-farm enforcement is a **partial UNIQUE index** on `registration_number_normalized WHERE NOT NULL` (mirrors the existing `vendor_profiles_business_slug_unique` pattern), plus a partial index on the review flag for the admin lane. Idempotent, no backfill.
- **`lib/vendor-registration-number.ts`** (pure, unit-tested) — `normalizeRegistrationNumber` (format-agnostic: a hyphenated TIN, DTI, or SEC number all normalize so equivalents collide; min length 5, clamped to 40), `parseRegistrationNumber` (empty / too_short / ok), and the shared vendor-facing collision copy.
- **Capture + enforcement** (`inline-docs-actions.ts` → `saveRegistrationNumberInline`, wired into the verification `DocsBody` on My Shop): the write claims the normalized identity on the vendor's OWN row. Collision handling is **RLS-safe by construction** — a vendor's session can only read its own row, so we rely on the DB unique constraint (SQLSTATE `23505`), never a pre-check SELECT. On collision the submission is NOT hard-blocked: the raw attempt is recorded, `registration_number_needs_review` is set TRUE (leaving `normalized` NULL so the index stays intact), and the vendor sees a clear "already registered — flagged for review" message. Re-submitting the same number is idempotent and clears a prior flag.
- **Required to submit** — `verificationSubmitMissing` gains an (optional, additive) `registrationNumberOnFile` input; threaded into all three call sites (inline `submitInlineForReview`, `/verify` `submitApplication`, and the My Shop `submitMissing` surface) so a shop can't reach `pending_review` (and its perk window) with no comparable identity on file. A collided number still counts as "on file" (it routes to admin review, not a block).
- **Admin surfacing** (`/admin/verify`): a soft-probe `regMap` (degrades on a pre-migration DB) feeds an amber "Duplicate reg #" badge + a Reg # line on each application card, so the existing manual verification review catches perk-farming before approval.

SPEC IMPACT: Adds a required, uniquely-indexed government registration number to the vendor verification flow (identity anti-abuse). Reflected in the code + this fragment; no locked SKU / pricing / schema-rename decision touched. Owner sign-off flagged in the PR for: (a) making the reg number REQUIRED to submit for verification, and (b) the soft "flag-for-review, don't hard-block" collision policy.
