## 2026-07-16 · feat(vendor): On-the-Day controllers are taxonomy-driven — kill the hardcoded 4-persona quad

The vendor "On the Day" console hardcoded its category controllers to exactly four personas (`CATEGORY_PILLS` = Photo/Video · Coordinator · Caterer · Band/DJ). That quad was invented taxonomy: `resolveDayOfConsoleKind` recognised only ~6 of the ~57 canonical `WeddingTile` keys a vendor actually stores in `services[]` (photo_video/editorial/livestream, catering/cake/stations, coordinator, music tiles) and collapsed **everything else** — florist, HMUA, photo booth, mobile bar, host/MC, fireworks, LED wall, souvenirs, stylist, bridal car, ~40 tiles — into a dead `'general'` fallback with generic copy.

This is PR-1 of the "On the Day → 4-step launcher" program (council verdict `Vendor_On_The_Day_App_Council_Verdict_2026-07-16.md`). It replaces the hardcode with a taxonomy-derived **controller-family + module** model, with no fake doors — the interactive configurator, launch route, and the counsel-gated capture/delivery modules land in later PRs behind their own schema + the DPO/NPC consent gate.

- `lib/vendor-dayof-modules.ts` (new): the pure module registry. Five controller **families** keyed to the 10 marketplace folders (`TILE_PARENT`) — coordinate/capture/serve/perform/setup — so every tile resolves to a real family (the ~40 old `'general'` tiles now land in `setup`), and a tile admin adds to the DB taxonomy inherits its folder's family for free. `resolveDayOfFamily` applies deterministic priority (coordinate > capture > serve > perform > setup) and narrows to the tiles booked on THIS event when a brief is present. A 12-module registry (`run_of_show`, `pax_headcount`, `delivery_handover`, `review_qr`, `live_reviews`, `qr_scanner`, `shot_list`, `setlist`, `issues_log`, `production_sheet`, `vendor_papic`, `guest_delivery`) with per-family defaults, availability, grant-delegation flags, and `counselGated` markers (papic + guest_delivery). `resolveModules` folds code defaults + a per-booking override into the on/off set (override plumbing arrives with `vendor_dayof_configs` in a later PR; today `override` is always null → pure code defaults).
- `lib/vendor-dayof-modules.test.ts` (new): 14 node:test cases — family mapping incl. the ex-`'general'` tiles, priority, event-tile narrowing, module availability/override/counsel-gate integrity.
- `app/vendor-dashboard/on-the-day/page.tsx`: deletes `CATEGORY_PILLS`; both pill blocks (full console + `CompactDayOf`) now render a `ModuleReadout` driven by `resolveDayOfFamily`/`resolveModules`. Counsel-gated modules render with a "Needs setup" badge rather than as silently-active tools. All existing live wiring (delivery handshake, RSVP headcount, editorial media, review QR, shot list, issues log, event-day gate) is unchanged. The legacy `resolveDayOfConsoleKind` still drives the console body sections until PR-2 folds them onto families.
