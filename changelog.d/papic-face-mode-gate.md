## 2026-07-22 · feat(papic): face-tag consent gate — Mode A/B, matcher enforcement, server-side minors gate, biometric observability scrub (flag-dark, DPO-gated)

Closes the One-Pool spec (`Papic_One_Pool_Model_Spec_2026-07-22.md` §3.4/§3.5) biometric blocker: `embedFaces()` ran UNCONDITIONALLY on every capture and POSTed 128-d face descriptors to the server, gated only by one global env var — so the moment face-sort was switched on, opt-outs / minors / bystanders / every generic-QR event would have been face-printed. Now face embedding fires only for consented (custom-QR opt-in) events, and never for the generic/shared QR.

- **Migration `20270831920476_papic_face_mode_gate_and_consent_copy_version.sql`**: `events.papic_face_mode TEXT NOT NULL DEFAULT 'mode_b' CHECK (…'mode_a'|'mode_b')` — `mode_b` (the safe default) computes/transmits NO face descriptor at all; only `mode_a` runs the embedder. Plus `guest_face_enrollments.consent_copy_version TEXT` (nullable) — consent evidence pinning WHAT disclosure was shown, bringing the per-event enroll path to parity with the account path's `ACCOUNT_FACE_CONSENT_VERSION`. Idempotent, no RLS change.
- **`lib/papic-face-mode.ts`** (new, isomorphic): `PapicFaceMode`, pure `resolveFaceMode`/`eventTypeForcesModeB` (fail-closed to `mode_b`; forces `mode_b` for `christening`/`debut`), async `resolvePapicFaceMode(client, eventId)`, and `FACE_CONSENT_COPY_VERSION`.
- **Client embed gate at all THREE capture call sites** — `papic-seat-capture.tsx` (photo + clip), `papic-guest-capture.tsx`, `booth-capture.tsx`: each now takes a server-resolved `faceMode` prop and does NOT call `embedFaces`/`embedClipFaces` unless `mode_a`. Threaded from all four render sites (`/papic/seat`, `/papic/guest`, patiktok `/booth`, and the inline camera on `app/[slug]/page.tsx`).
- **CI guardrail `lib/papic-face-mode-gate.test.ts`**: source-scans the three capture files and FAILS the build if any embed call lacks a Mode-A gate; also asserts both server matchers enforce the control + mode.
- **Server matcher enforcement**: `autoTagCapture` (`lib/face-match.ts`) and `matchPatiktokFace` (patiktok `actions.ts`) now fail-closed on BOTH `isDataPrivacyControlActive('face_enrollment')` (previously ZERO runtime callers — a paper record) AND resolved `mode_a`, so a crafted POST of vectors to a Mode-B / control-off event is dropped server-side.
- **Server-side minors gate**: `enrollGuestFace` now hard-requires `age_affirmation` alongside `biometric_consent` (was client-only; a crafted POST could enrol a minor) — this is also the custom-QR enroll path (cookie-session). `submitRsvp` already enforced it. Both enroll paths now stamp `consent_copy_version`. `christening`/`debut` are forced to `mode_b` (no embedding) via `resolveFaceMode`.
- **Biometric observability scrub `lib/observability-scrub.ts`** (new): fail-closed Sentry `beforeSend` denylist (key-based + positional descriptor-array shape) wired into the browser (`deferred-observability.tsx`, composed with the existing PII scrub), Node (`sentry.server.config.ts`), and edge (`sentry.edge.config.ts`) configs so a face vector can never reach Sentry or a captured request/action payload. Asserted in `lib/observability-scrub.test.ts`.
- **On-device matching is intentionally NOT built** (spec §3.4 step 4) — it is a DPO deliverable; a `// TODO(dpo)` posture is documented in the spec, and Mode A must not GA on real data until the DPO ruling clears.

## 2026-07-22 · fix(papic): close the ENROLLMENT face-mode hole the red-team found (the gate's own guarantee)

Adversarial red-team found the PR gated the three CANDID-capture surfaces but left the ENROLLMENT path ungated — so `mode_b` (incl. forced-`mode_b` christening/debut) still computed a descriptor client-side AND persisted whatever the client POSTed, falsifying the migration's "no face descriptor computed/transmitted/stored" guarantee and defeating the minor-honoree protection. Now enforced end-to-end.

- **`lib/papic-face-mode.ts` — new pure guard `faceVectorForMode(mode, candidate, vectorModel)`**: returns the descriptor + model only in `mode_a`, else HARD-NULLS both. Isomorphic, dependency-free (`vectorModel` injected).
- **`selfie-capture.tsx` — client embed gate**: now takes a `faceMode` prop (fail-closed default `mode_b`) and does NOT call `embedSingleFace` unless `mode_a`; the selfie still uploads as a profile photo, no descriptor is computed/transmitted. Threaded via `DayOfFaceEnroll` → all render sites: `app/[slug]/page.tsx` (RSVP `RsvpWidget` + day-of enroll — resolved once with `resolvePapicFaceMode`), `app/[slug]/hub/page.tsx` (resolved when face-enroll is offered), and the inline `papic-guest-capture.tsx` camera (passes its existing `faceMode`).
- **BLOCKER — server-side write guard in BOTH enrollment actions**: `submitRsvp` (`app/[slug]/actions.ts`) and `enrollGuestFace` (`app/papic/face-enroll-actions.ts`) now resolve the effective mode server-side (`resolvePapicFaceMode` + christening/debut forcing) and route every descriptor through `faceVectorForMode` before the `guest_face_enrollments` insert — so a crafted POST carrying `selfie_vector(s)` on a `mode_b`/forced-`mode_b` event can NEVER persist a biometric, even with `biometric_consent=1`+`age_affirmation=1`. Selfie image + consent row are still written (profile-photo / day-of features preserved); only the vector is dropped. This closes the christening/debut minor leak.
- **CI guardrail `lib/papic-face-mode-gate.test.ts` un-blinded**: `EMBED_CALL_RE` broadened to match ANY embedder (`embed…Face…(` — `embedFaces`/`embedClipFaces`/`embedSingleFace`); `selfie-capture.tsx` added to `CAPTURE_FILES`; a new discovery test walks ALL of `app/` so every embedder call site must be a gated `CAPTURE_FILE` or a documented `EMBED_EXEMPT_FILE` (the homepage demo — per-session self-consent, ephemeral relay, never persisted, no event binding — is the one explicit, commented exemption); new source-scan guards assert both enrollment actions keep `resolvePapicFaceMode` + `faceVectorForMode`. Verified RED when the selfie gate is removed.
- **New tests `lib/papic-face-mode.test.ts`**: `faceVectorForMode` stores the vector only in `mode_a`, hard-nulls it in `mode_b` even when the payload carries one, drops it for forced-`mode_b` christening/debut with a consented payload, nulls every shot in a multi-shot enroll, and composes with async `resolvePapicFaceMode` for a christening row that (falsely) says `mode_a`.
