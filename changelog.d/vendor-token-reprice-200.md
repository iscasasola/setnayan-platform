- **`supabase/migrations/20270728100000_vendor_token_pack_reprice_200.sql`** — restructures the `vendor_billing_catalog` token packs to a flat **₱200/token** anchored at ₱1,000 = 5 tokens. **sku_code decision — retire + insert (not rename):** the old bottom pack `vendor_token_pack_4` (4 tokens) is RETIRED via `is_active=false`, and a new `vendor_token_pack_5` (5 tokens, ₱1,000, `display_order` 30) is upserted; `vendor_token_pack_10/25/50/100` are repriced to ₱2,000/5,000/10,000/20,000. This is safe because (a) `vendor_billing_catalog.is_active` is a first-class retire flag honored by BOTH the catalog reader (`lib/v2-catalog.ts` `fetchV2VendorCatalog()` filters `.eq('is_active', true)`) AND the purchase RPC (`create_vendor_token_purchase()` looks up `… AND is_active = TRUE`, raising `INVALID_PACK` otherwise), and (b) `vendor_token_purchases.pack_sku_code` is plain TEXT with NO FK to the catalog and the purchase row snapshots `token_count` + `amount_php` at buy time — so retiring pack_4 leaves every historical purchase intact and yields a clean, self-describing `_5` sku_code rather than a `_4` row granting 5 tokens. Idempotent (`ON CONFLICT` upsert + guarded `UPDATE`s). `token_grant_count` preserved on repriced packs; `price_php` is whole pesos. `getVendorPrices()` derives ₱/token as `price_php ÷ token_grant_count` = ₱200 for every pack, so `/vendors`, `/pricing`, and the admin/token surfaces pick up ₱200 with no code change.
