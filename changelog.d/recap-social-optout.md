## 2026-07-16 · feat(privacy): couple opt-out for Setnayan re-posting a published recap + private-site guard

Social-sharing follow-through **#2**. When a couple publishes their event recap, Setnayan MAY feature it on Setnayan's OWN Facebook / Instagram through the existing social pipeline (`lib/social/recap-post.ts` compose → `lib/social/flush.ts` dispatch). That path had **no `landing_page_visibility` check and no per-event opt-out** — inert today only because Meta creds are env-dark. This lands the consent gate BEFORE those creds are ever armed. Default stays opt-IN ("everything public initially, then they can set it private").

- **Schema** — `supabase/migrations/20270812578060_events_recap_social_optout_at.sql`: one nullable `events.recap_social_optout_at timestamptz` (NULL = allowed = default). A single per-event opt-out, NOT a `marketing_share_consents` grant and explicitly NOT `public_summary_consent_at` (that gates the `/realstories` index — a different surface). RLS on `public.events` (`couple_can_update_event`) already scopes the write; no new policy.
- **Shared gate** — `isRecapSocialShareAllowed(admin, eventId)` in `lib/social/recap-post.ts`: allowed only when `recap_social_optout_at IS NULL` AND `resolveEffectiveVisibility(event) === 'public'`. Fail-CLOSED on any read error so a transient DB hiccup can never leak a private recap.
- **Compose gate** — `composeRecapSocialPost` returns before rendering/inserting when the gate fails (checked right after the per-feature autopost switch, before the expensive card render).
- **Dispatch gate** — the `dispatchDuePosts` loop pulls (`scheduled → pulled`) any `event_recap` row whose gate now fails, before the publish claim — catching an opt-out or a visibility flip that happened AFTER compose. A private/unlisted couple is never dispatched (unanimous council red line #2).
- **UI** — recap manager (`dashboard/[eventId]/studio/papic/recap/`): new "Featuring on Setnayan's socials" section with a switch ("Let Setnayan feature this recap on our Facebook & Instagram", checked by default). Off stamps `recap_social_optout_at = now()`; on clears it, via `setRecapSocialFeatureAllowed`. Copy notes it only posts after the event and can be turned off anytime; an already-live post is removed by the existing admin Social Queue take-down (24h SLA), not rebuilt here.
