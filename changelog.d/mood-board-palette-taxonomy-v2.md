## 2026-07-08 · feat(mood-board): palette taxonomy v2 — split wedding-party keys + parents/Nikah keys + room dressing

Owner-locked 2026-07-08. The Wedding Party palette SPLITS into four specific role keys (`maid_of_honor` covers maid + matron · `best_man` · `bridesmaids` · `groomsmen`); the existing `wedding_party` set STAYS as the shared FALLBACK. Two new keys land: `parents_immediate_family` (both sides' parents + immediate family — supersedes the dormant, never-persisted `vip_family` palette key) and `muslim_principals` (the Nikah cast — wali/witness/imam/wakil — a REAL attire key now; the old `PaletteKey` exclusion is lifted). It surfaces only for muslim weddings via the existing Nikah role set, so non-muslim boards are unchanged. No DB migration — `role_palette` is JSONB.

- `lib/mood-board.ts`: `PaletteKey` gains the split + parents keys and lifts the muslim_principals exclusion; `PALETTE_ORDER` / `PALETTE_LIMITS` / `DEFAULT_PALETTE_SUGGESTIONS` / `ROLE_FAMILY_KEYS` extended (compiler-forced via the exhaustive `Record<PaletteKey,…>`). `sanitizeRolePalette` stays non-destructive and now preserves/validates a new `room_dressing` override block. New pure resolvers: `paletteKeyForRole(role)` · `resolveAttirePaletteColor(role, palette, sideColor)` (STRICT chain: specific key → wedding_party → bride/groom SIDE color → kit default `null`) · `sideAttireColor(palette, side)` · `resolveRoomDressing(palette)` (linens/chairs/florals/lighting_warmth DERIVED from the reception palette, per-field overridable). `getPrimaryColor` accepts the role GROUP again and normalizes `vip_family` → `parents_immediate_family`.
- `lib/seating-3d.ts` `resolvePaletteFromRoles`: applies room-dressing overrides (linen → table material, lighting warmth → ambient) — surgically, so a palette with no `room_dressing` returns the byte-identical pre-taxonomy result.
- `lib/site-palette.ts`: guarded the arbitrary-key palette pool walks against the non-array `room_dressing` value (would have crashed `.forEach` on any event that saves overrides).
- 3D attire wiring — a specific role key wins, else `wedding_party`, else side, else neutral: `seating/lab/page.tsx` (per-role, per-side via `resolveAttirePaletteColor`), `plan3d-scene.tsx` + `plan3d-demo-actions.ts` (demo now carries `role` and routes the same chain).
- Editor UI (`mood-board/page.tsx` + `palette-editor.tsx`): each new section renders ONLY when the guest list contains that role (per-role detection via `paletteKeyForRole` over primary + extra roles; muslim_principals rides the existing Nikah gate); a collapsed "Advanced · Room dressing" panel previews the derived colors and lets couples override per field.
- Tests: new `lib/mood-board.test.ts` (24 cases) — legacy `wedding_party`-only round-trip + resolve identity, new-key round-trip/clamp, room_dressing preserve/validate, `getPrimaryColor` normalization, the full `paletteKeyForRole` table, the STRICT attire chain, and derived-vs-override room dressing.

BACKWARD COMPAT: existing saved `role_palette` JSONB is untouched — sanitize preserves legacy keys, and a couple who set only `wedding_party` sees identical GOWN attire. NOTE (surfaced for owner): the STRICT chain now also lets the MALE entourage (best_man/groomsmen, and the groom when his own key is empty) fall back to `wedding_party` — previously they took the groom color / charcoal default. This is the taxonomy's stated intent ("wedding party = bridesmaids · groomsmen · MoH · best man"), but it changes suit color under a wedding_party-only palette. Empty-palette gowns/suits now use the kit's default cloth instead of the old blush/charcoal hardcodes (matches the demo scene, which already did this).
