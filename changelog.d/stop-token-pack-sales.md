## 2026-07-22 · feat(vendor-billing): retire vendor token packs — and make answering an inquiry free first, so no paid vendor is stranded

Owner retired tokens (2026-07-21) and wants pack sales stopped. The delicate part: answering a couple's inquiry BURNED a token, so naively retiring the packs would strand token-less vendors. This lands in the safe order — free the answer, THEN stop the sale.

**STEP 1 — the current truth (before this change).** The live accept path is `chat-actions.ts` `acceptInquiry` → the `unlock_vendor_event` RPC (the parallel `unlock_vendor_event_hold` path is gated by `NEXT_PUBLIC_LEAD_TOKEN_HOLD_ENABLED`, default OFF, and is dormant). The newest live body of the RPC (migration `20270819553697`, verbatim from `20270818135217` founder_seats) burned `regions.burn_band` (1-3) tokens for every NEW `(vendor,event)` unlock on **verified · solo · pro · enterprise** (verified was moved onto the burning path at `20270307985604`, so the old "verified answers free" carve-out is gone), and raised `INSUFFICIENT_WALLET_BALANCES` — rolling the whole tx back — when the answering member had no balance. Free tier is blocked outright (`TIER_FREE_NO_INAPP`); a founder-seat event is comped (0 tokens); re-accepting an already-unlocked pair is free. Token sources: **packs** (being retired), admin grants, and subscription bundle grants. So a paid vendor whose only token source was packs could be left unable to answer at all.

**STEP 2 — answering is now free.** Migration `20270909586177_free_inquiry_answer_no_token_gate.sql` re-issues `unlock_vendor_event` VERBATIM from the live body with one change: the answer's token cost is pinned to zero (`v_tokens := 0`) before the unlock is recorded, so the `IF v_paid AND v_tokens > 0` debit branch is unreachable — no `consume_*` runs, and `INSUFFICIENT_WALLET_BALANCES` can no longer fire. Every gate is preserved: answering-member check (`FORBIDDEN`), free-tier block (`TIER_FREE_NO_INAPP`), the verified ≤10-new-unlocks/rolling-week throttle (`VERIFIED_WEEKLY_LIMIT`, a tier limit not a purchase gate), idempotent re-accept, and founder comp. Wallet / bundle-grant / burn plumbing is left intact — reverting this migration restores burn-on-answer.

**STEP 3 — packs retired (sale stopped).** Migration `20270910266901_retire_vendor_token_packs.sql` sets `is_active=false` on `vendor_token_pack_5/10/25/50/100` (rows kept — `vendor_token_purchases.pack_sku_code` snapshots stay valid; idempotent). This is the honored path: `fetchV2VendorCatalog()` filters `is_active=true` (the Plan & tokens hub's `BuyTokensCta` falls to its existing empty state — no pack renders or can be started) and `create_vendor_token_purchase` guards `is_active=TRUE` (raises `INVALID_PACK`). Guard surfaces reconciled: `lib/llms-price-fixture.ts` drops the token-only figures `₱200 / ₱5,000 / ₱20,000` and keeps `₱1,000` (Solo 28-day) · `₱2,000` (Stories per-day cap) · `₱10,000` (Solo annual); `public/llms.txt` replaces the "Vendor tokens" pack table + the flat-₱200-per-answer line with an "Answering couples (included free)" section and drops the one stale "token-to-unlock" clause, so the body ₱-set matches the fixture both ways. The wallet / bundle / burn economy is left dormant, not torn out.

Tests — new `lib/free-inquiry-answer.test.ts` (statically proves the newest `unlock_vendor_event` pins the answer cost to zero ahead of the unlock + debit guard, and keeps the free-tier block). `lib/llms-price-drift.test.ts` reconciled and green.
