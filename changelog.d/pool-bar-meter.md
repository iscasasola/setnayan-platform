## 2026-07-23 · feat(papic): host capture-pool meter card — read-only, flag-dark (build ③ PR-1)

WAVE-2 LANE F · study § 3.4's "parity pool-meter card" for the couple dashboard. The one-pool machinery has been fully shipped for weeks (`papic_event_point_grants` ledger, atomic `papic_reserve_event_points`, the SECURITY DEFINER `papic_event_pool_status` reader) — but `fetchEventPoolStatus` had exactly ONE caller (the seat capture path) and no host surface anywhere: a couple had no way to see their own pool draining. This PR adds the READ-ONLY meter; deliberately NO money wiring (no top-up doorway, no purchase copy — the charge path is a later, separately-supervised PR; `PAPIC_GUEST_TOPUP` stays `is_active=FALSE`).

**New:**

- `lib/papic-pool-meter.ts` — host display shaping over the SHIPPED reader (no new data path): `shapeHostPoolMeter` (pure — clamps, pct math, `ok`/`low`/`exhausted` level with the amber line strictly below `POOL_METER_LOW_PCT = 10`% remaining, judged on raw ratios so a 9.6%-remaining pool can't hide behind rounding) + `fetchHostPoolMeter` composing `fetchEventPoolStatus` (same admin-client requirement, same graceful-degrade: any RPC problem → null → card hidden, never a broken page). Returns null when the pool doesn't apply (no flat pass, no grants — the unlimited legacy posture).
- `lib/papic-pool-bar-flag.ts` — `papicPoolBarEnabled()` reading `NEXT_PUBLIC_PAPIC_POOL_BAR` (default OFF, documented in `.env.example`; build-time inlined, flip requires redeploy — fails safe).
- `app/dashboard/[eventId]/studio/papic/_components/host-pool-meter-card.tsx` — server component, self-gating like its sibling cards: flag → membership (event row read under the VIEWER'S RLS session; non-members resolve nothing) → pool-applies. Renders remaining/total, a meter bar (`role="meter"` + aria values), amber low state, red exhausted state, "includes N added points" when ledger grants exist, and the point-currency line derived from `PAPIC_POINTS_PER_PHOTO`/`PAPIC_POINTS_PER_CLIP` (never hardcoded — the 7-pt clip constant is owner-volatile). Mounted on `studio/papic` directly under the "Your cameras" section it meters — deliberately away from the add-on region, whose surroundings PR #3581 (PoolGalleryCard) was concurrently editing while blocked on a failed required check; if #3581 lands after this, both hunks merge cleanly (disjoint line ranges).
- `lib/papic-pool-meter.test.ts` — 13 unit tests on the meter math: absence→null, degenerate zero-total→null (never a false red bar), grant-only events (base 0, total==SUM(grants)), the strict 10% boundary (exactly 10% = ok, one point under = low), raw-ratio-vs-rounded-pct, exhaustion, and over-total clamps.

**Zero schema.** No migration, no RLS change, no touch on `papic_reserve_event_points`/`papic_event_pool_status`. No guest-site file touched (`app/[slug]/*` reserved for the open-browse rebuild). Pricing/checkout surfaces untouched — including `extra-cameras-picker.tsx`, whose stale "Mini/Ltd/Unli per-day" header doc + misleading `budgetLine()` null-branch ("No limit · archived to your Drive" for the pool-metered Papic One) are known and left for the money PR's ride-along fix (study § 3.4).

**Papic One caveat (owner 2026-07-23):** Papic One camera points are conceptually DEDICATED per camera, but the shipped ledger pools them event-wide — `papic_event_point_grants` has no seat scope and `papic_event_pool_usage` is one per-event counter with no seat attribution, so a per-camera "dedicated remaining" cannot be derived from what exists. The meter truthfully reports the ONE shared pool the reserve RPC actually enforces; modelling dedication is a ledger change owned by a later supervised PR.
