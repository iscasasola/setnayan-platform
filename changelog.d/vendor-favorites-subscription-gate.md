## 2026-07-18 · feat(vendor): favorites are subscription-gated — lapse hides them, re-subscribe restores

Owner-locked 2026-07-18 (corpus DECISION_LOG): a vendor may be SEEN as a favorite / saved only while they hold an ACTIVE PAID subscription (Solo ₱999+ / Pro / Enterprise / Custom). When a subscription lapses to a free tier the vendor's favorite is hidden from every couple/coordinator saved list — immediately, no grace — but the underlying favorite ROW is never deleted, so it reappears automatically on re-subscribe ("will only return if subscribed again"). The favoriter sees the card silently disappear; the vendor stays discoverable via normal search score.

The lapse is silent on the vendor side too — owner reversed the earlier "email the vendor with the count" answer (2026-07-18: "no email needed"), so no nudge email/template is built. This is the complete feature: the visibility gate, behind a flag defaulted OFF.

- **New `apps/web/lib/vendor-favorite-gate.ts`** — the single source of truth. `vendorHoldsActivePaidSub({tier_state, tier_expires_at})` = `isTierAtLeast(tier,'solo') && (tier_expires_at is null OR > now)`, mirroring `enterprise-vendor-gate.ts` (expiry checked explicitly because lapse is login-driven via `sweep_vendor_tier_expiry`). `filterFavoritableVendorIds(admin, ids)` returns the favoritable subset; **hiding is a READ-TIME FILTER, never a write** — lapse does nothing to favorites data, so restore is a no-op and there is no cron/batch leak window. Fail-OPEN on a read error (a DB blip must not blank a couple's saved list; the gate is a monetization nudge, not a security boundary). Kept free of `import 'server-only'` (pure predicate + passed-in client) so it is unit-testable.
- **Gated the two Library "Saved Vendors" loaders** — `library/_data/saved-vendors.ts` (own-event `event_vendors` picks) and `library/_data/attended-vendors.ts` (`guest_saved_vendors` attended bookmarks) now filter their vendor ids through the gate before hydration. `hydrateVendorCards` (shared with the event-credits list) is deliberately untouched — the gate lives in the favorites loaders, not the shared hydrator. The `VendorsTab` empty state already handles a fully-filtered list.
- **Vendor-side "Saved" COUNT deliberately NOT gated** — it reads the raw tables via the `count_saves_for_vendor` SECURITY DEFINER RPC, so the vendor keeps seeing the TRUE, preserved count. No change needed; asserted by the design.
- **Coordinators / marketplace pin / auto-favorite** — no code: coordinators act on the same `event_vendors` picks (covered by `saved-vendors.ts`); the marketplace "Favorites-first" sort tier and `auto_favorite`/`rebook_favorite` are spec-only and do not exist in the codebase yet, so there is nothing to gate there.
- **Activation flag** `VENDOR_FAVORITES_SUBSCRIPTION_GATE` (server-only, default OFF), documented in `.env.example`. MUST stay OFF during free-during-launch; flip to 'true' only once paid vendor subscriptions are mandatory (owner sign-off on activation timing pending).
- **Guards** (`lib/vendor-favorite-gate.test.ts`): predicate truth-table (paid×expiry, free tiers, lapsed tier, flag default-OFF/only-'true') + a source-scan guard (same shape as `llms-price-drift.test.ts`) asserting both Library favorites loaders call `filterFavoritableVendorIds` — a new favorites surface can't silently skip the gate.

No migration — the gate reads existing `vendor_profiles.tier_state` / `tier_expires_at`.
