## 2026-07-12 · feat(events): date-anchor model — foundation layer (anchor attributes + derivation engine)

The un-gated foundation of the date-anchor model (owner "build it" · design in the spec corpus at `Event_Anchor_Minimalist_Setup_Design_2026-07-12.md` + `Event_Anchor_Model_Council_Verdict_2026-07-12.md`). Anchors are the primitive that turns one-off events into a recurring family relationship — store an anchor once, derive every future occurrence (next birthday, the Nth anniversary, the 18th debut, the milestone ladder) at read time. Per the council's Conflict-E ruling the per-type defaults ship as a PURE TS MAP (not a vocab/taxonomy rewrite); recurrence is a suggestion flag (no RRULE engine, no auto-created rows — the model layer stays cron-free/Rule-1 deterministic).

- **Migration `20270731291090_event_anchor_foundation.sql`** — adds four per-event columns to `events`: `anchor_kind` (CHECK'd enum: person_birthdate · union_date · expected_due_date · fixed_date · date_range · calendar_holiday · none), `anchor_date` DATE, `anchor_origin` (CHECK'd to POSITIVE origins only — wedding/relationship/milestone/matters — the DB itself refuses a memorial/death origin, honoring the burial retirement so generalized anniversaries can't backdoor babang-luksa), and `recurs` BOOLEAN. No new tables → RLS unchanged. Idempotent.
- **`lib/event-anchor.ts`** — the deterministic, dependency-free derivation engine: `ANCHOR_BY_TYPE` (the authored per-type default map, source of truth) + `anchorForType()`; the PH milestone ladder `1 · 7 · 18(F)/21(M) · 60` (`nextMilestone`, sex-aware, both offered when unknown); recurrence + anniversary math (`nextOccurrence`, `nextAnniversary`, `nextBirthday`) hardened across year and Feb-29 boundaries; and the authored lead-time ladder (§ 4c) `leadTimeFor()` + `nudgePlan()` with the December override. Every function takes dates as arguments and stores nothing — the engine holds no PII.
- **`lib/event-anchor.test.ts`** — 29 unit tests (ladder per sex, leap-day recurrence, Nth anniversary, the December override, date-helper clamping). All green.
- **Wiring** — `createWeddingEvent` (live path) and `buildGenericEventInsert` (flag-dark generic path) now stamp `anchor_kind` at insert from the type map (wedding → `none`, it PRODUCES a union date; birthday/debut → `person_birthdate`; etc.). No behavior change to any existing surface.

**Deliberately NOT in this PR (owner-locked counsel gate):** the dependent People layer stores minors' birthdates for up to 18 years (RA 10173) — it is counsel-gated (PR-D). The milestone helpers compute over a caller-supplied birthdate so the engine is ready without holding any PII. The creation-flow UI (per-type anchor question, the 4-page creator), the wedding→anniversary link, and the Year-view moments surface are the next PRs (PR-A / PR-F).
