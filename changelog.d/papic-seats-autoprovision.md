## 2026-06-22 · feat(papic): auto-provision the 5 seats on payment approval (no manual "set up your seats" step)

Owner-locked rule: a paid feature must be READY with no manual activate step once the order is admin-approved — the activation hook IS that approved moment. PAPIC_SEATS (₱2,999 · "Turn five friends into your photo crew") was the exception: its activation hook only made any free-sampler photos permanent + cancelled the expiry-warning emails, and the couple still had to tap a "Set up your 5 seats" button on `/crew` to materialize the seats. Now the seats exist the instant the Setnayan team approves the order.

- `lib/papic-seats.ts` — NEW `provisionPapicSeatsAdmin(admin, eventId)`. Service-role, idempotent TOP-UP: reads the existing paid (`is_free_sampler = false`) seat_index set and inserts ONLY the missing indexes in 1..5 (`upsert` with `onConflict: 'event_id,seat_index', ignoreDuplicates: true` — the UNIQUE (event_id, seat_index) backstop). Mirrors the SECURITY DEFINER `papic_provision_seats()` RPC's insert exactly (dense 1..5, `sku_code 'PAPIC_SEATS'`, fresh `generateSeatClaimToken()`), but skips the RPC's `auth.uid()` guard so it can run under the admin client (the RPC raises `28000 not authenticated` when `auth.uid()` is NULL, so the existing couple-facing action can't be reused from the hook). Best-effort/non-fatal (returns 0 on any error) so a write failure can never roll back the approval — same posture as `makeSamplerPermanent`.
- `lib/sku-activation.ts` — the `PAPIC_SEATS` activation hook now ALSO calls `provisionPapicSeatsAdmin(ctx.admin, ctx.eventId)`, fault-isolated in its own try/catch so it can't starve (or be starved by) the sampler-permanence step. Honors the file's "every hook is non-fatal + idempotent" contract; the lint guard's required `PAPIC_SEATS` + `makeSamplerPermanent` tokens are preserved.
- `app/dashboard/[eventId]/studio/papic/crew/page.tsx` — the paid-crew empty state is reframed from a REQUIRED first step ("Set up your 5 seats") into an idempotent TOP-UP fallback ("Top up / Fill in your 5 seats") — seats normally exist post-approval; the button now only fills any shortfall. The form already calls the idempotent `provisionPapicSeats` action (ON CONFLICT DO NOTHING via the RPC), so no behavioral change beyond copy.
- `app/dashboard/[eventId]/launch/page.tsx` (day-of launch hub) — when `ownsPapic` is true the Papic card now surfaces a "ready — share these 5 seat links" affordance (dynamic blurb + "Share the 5 links" CTA) instead of implying setup is still pending. Reuses the existing `eventPapicSeatsActive` ownership read; no new query.

Idempotency guaranteed three ways: (1) read-then-insert-missing-only in the helper, (2) `ignoreDuplicates` on the upsert, (3) the DB-level UNIQUE (event_id, seat_index). Safe to re-run on a re-approved order, on a bundle (MEDIA_PACK) buyer whose hook fans through PAPIC_SEATS, or after a couple already self-served via /crew.
