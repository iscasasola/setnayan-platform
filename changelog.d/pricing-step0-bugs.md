## 2026-07-21 · fix(pricing): paid Live Studio events were shown a buy button — v1 SKU codes matched against a V2 `orders.service_key`

Fallout from the admin-pricing cleanup audit (5-seat council + 22 adversarial verifiers · verdict at `~/Documents/Claude/Projects/Setnayan/Admin_Pricing_Cleanup_Council_Verdict_2026-07-21.md`). The audit was asked "what old pricing can we delete?" and the answer was **almost none** — but it surfaced three live defects on the way. This PR is those three, and deletes no catalog row.

**1. `lib/add-on-stats.ts` — `ADD_ON_SKU_MAP.panood` listed seven v1 codes that can never match.** The map is `.in('service_key', skus)`-matched against `orders.service_key`, which has stored **V2** `platform_retail_catalog_v2.service_code` values since the catalog cutover — but `panood` still named `panood_daily_broadcast`, `panood_annual_streaming`, `ai_video_highlight_60s`, `ai_edited_highlight_3min`, `custom_monogram_pack`, `broadcast_style_pack`, `pro_camera_bridge_addon`. Zero of the seven exist as a `service_key`. This was read as a stats-only map, but `lib/add-on-state.ts:99` imports it too and feeds it into `resolveAddOnState`, which decides `'launch'` vs `'add'` — so the stale codes weren't under-counting a badge, they were **locking a paying couple out of a feature they bought**. `FEATURE_BUNDLE_SKU.panood` already held the correct `PANOOD_SYSTEM`, but it only resolves *granting bundles*; an à-la-carte order never matched. Measured in prod: event `fba4a392…` holds **2 paid `PANOOD_SYSTEM` orders** and was being served a buy button instead of its control room. Now `['PANOOD_SYSTEM', 'PANOOD_SYSTEM_MOBILE']` — both rungs of the 2026-07-20 Live Studio repackaging.

**2. `app/api/v1/billing/initialize-maya/route.ts` — `readSkuPrice` ignored `is_active`.** Its sibling `readBundlePrice` filters `.eq('is_active', true)` (added 2026-06-29 so the deactivated GUIDED_PACK/MEDIA_PACK can't be billed), but the à-la-carte reader never got the same guard — so a **retired** `service_code` still resolved to a real `retail_price_php` and could be charged through the gateway. Asymmetry closed: retired code → no row → `null` → caller 400s, matching the bundle path's fail-closed contract.

**3. `lib/sku-catalog.ts` — the v1 mirror claimed 19 active SKUs; prod has 1.** Eighteen rows retired in `service_catalog` on 2026-05-16/05-28 (`monogram_hero_upgrade`, `pro_widget_schedule`, both Panood ladders, all three Paparazzi packs, both AI-highlight SKUs, the five `tool_*_weekly` integrations, `all_tools_unlock_annual`, `vendor_pro_weekly`, and the two `vendor_verification_*` renewal rows) were still `isActive: true` here, so `findSku()` handed callers live-looking prices for SKUs nobody can buy. Flipped to match prod; `vendor_verification_initial` is the one row that stays active. Header now marks the file LEGACY and points at `platform_retail_catalog_v2` / `lib/v2-catalog.ts`.

⚠ **Scoping note — this does NOT fix the `/site-editor` price leak, contrary to the audit's own claim.** The council asserted the flag flip would stop `/site-editor` rendering retired ₱1,999 / ₱999 to couples. It doesn't: `ProCard` falls back to a hardcoded `fallbackPrice` string on a `findSku` miss, and those strings (`"₱1,999"`, `"₱999"`) are byte-identical to the `priceCentavos` they replace (`199900`, `99900`), so the flip is **visually inert** there. Both cards still advertise retired SKUs at retired prices. Left alone deliberately — whether those two cards are still offered at all is a product decision, filed as an owner sign-off item in the verdict, not something to silently reprice on a customer-facing surface. The other two live `findSku` consumers degrade cleanly (`upcoming-items.ts:499` falls back to `row.description`; prod has zero subscription-expiry rows carrying a legacy `service_key`).

Only the three defects above are touched. No catalog rows deleted, no migration, no schema change — the audit's finding was that the ~63 inactive catalog rows are already invisible to customers and are re-seeded by `20260516000000` with `ON CONFLICT DO UPDATE`, so deleting them desyncs prod on every `db reset`. The admin-surface decluttering (a `<details>` "Show N retired" toggle, retiring the structurally-broken Add-ons tab) is a separate follow-up PR.
