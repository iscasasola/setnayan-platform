## 2026-07-23 · feat(chat): off-platform-contact filter — mask + flag couple↔vendor disintermediation (flag-dark)

Owner: "on chat, we would like to block or flag messages containing contact numbers, facebook account, viber, messenger, whatsapp, blue app, purple app, ig, email, or other means to communicate outside — and this will be flagged." Chosen enforcement (owner-confirmed): **mask + flag**, **both directions** (couple + vendor).

**Why.** Setnayan's economy assumes the deal stays ON the platform (vendor booking fee + vendor subscription + couple SKUs). The biggest leak is disintermediation over couple↔vendor chat — one party drops a number / email / "add me on Viber" and the relationship walks off Setnayan. This adds a deterministic (no-LLM, ₱0/scan) filter that MASKS the actual contact payload in the delivered message and records the ORIGINAL to an admin review queue.

**Detection core — `apps/web/lib/chat-contact-filter.ts` (pure, 17 unit tests).** `scanForContactInfo(body)` → `{ hasHit, hits[], masked, categories[] }`. Two buckets: HARD tokens carrying a payload — phone (≥9-digit phone-shaped run, so ₱12,999 / 150 pax / 2026 pass clean), email (+ `(at)/(dot)` obfuscation), social/messaging URLs (`facebook.com`, `wa.me/…`, `m.me/…`, `t.me/…`, `viber`, `instagram`, …), `@handle`; and SOFT intent signals with no payload — bare app names (`facebook fb messenger viber whatsapp telegram instagram insta ig snapchat tiktok …`, word-boundaried so "ig" won't fire inside "big"), colour-coded euphemisms (**blue/purple/green/pink app**), and solicitations ("add/message/text me on…", "my number is", "off platform"). `MASK_CATEGORIES` (one constant) controls which hits get blanked — defaults to all; move the soft categories out to make them flag-only.

**Enforcement — wired into the ONE shared chokepoint `sendChatMessageCore` (`apps/web/lib/chat-send.ts`).** Covers web server action AND the native JSON send endpoint. After all gates pass: scan `trimmed`, deliver `masked` as the message body, pass `masked` to the notification/email (payload never leaks via push), and record the original to `chat_message_flags` off the request path via `after()` (best-effort — a flag-write failure never blocks the send). System/bot messages insert via the service-role client and never reach this core, so they're exempt.

**Flag queue — migration `20270920573307_chat_message_flags.sql` · METADATA ONLY.** Admin-only table (`is_admin()` RLS · read + update), written by the service-role client (no authenticated INSERT policy). Stores **categories + hit_count + sender_role + context ids — NEVER the message text**, because the owner-locked admin-account-access model (2026-06-22 · `lint-admin-chat-guard`) forbids Setnayan staff from reading couple↔vendor chat bodies (published trust promise). The mask is the enforcement (needs no admin reading); the flag is a pure abuse signal (spot a repeat off-platform-pusher by category + frequency, not content). Admin surface `/admin/chat-flags` (page + actions, mirrors `/admin/user-reports`) shows the signal only, with an open/reviewed/dismissed filter + nav entry.

**Ships DARK.** Gated by server flag `CHAT_CONTACT_FILTER_ENABLED` (default OFF · `apps/web/lib/chat-contact-filter-flag.ts` + `.env.example`). OFF ⇒ send path byte-identical to before (no scan, no mask, no flag). Owner flips on after pushing the migration and eyeballing the queue on a test thread.

Judgment calls surfaced to owner: (1) the flag is metadata-only to honour the 2026-06-22 admin-no-chat-read lock — if the owner wants moderators to SEE the flagged text, that's a deliberate reversal of that locked promise and must route through the consent/takeover model, not this queue; (2) app-name mentions (e.g. "Instagram") are masked by default (a false positive hides one word + flags — never blocks the whole message); (3) scanning is text-only (a number inside an attached business-card image is an OCR follow-up).

Tests: `chat-contact-filter.test.ts` (17), `chat-contact-filter-flag.test.ts` (3). Typecheck + lint clean.
